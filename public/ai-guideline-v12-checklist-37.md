## はじめに

「AI事業者ガイドライン v1.2 に対応しろ」

社内のAI推進PJでそう言われて、原文を読んだ。正直、最初の感想は「多い」だった。

v1.1の25要件から **37要件に増えている**。しかもv1.2で追加された12要件は、Claude CodeやCursorみたいなAIエージェントの普及を前提にした内容で、「緊急停止メカニズム」とか「エージェンティックAIの安全設計」とか、なかなかハードな要件が並んでいる。

で、ツール探しを始めたんだけど、**日本のAI規制に対応したOSSが見当たらない**。海外のガードレールツール（Guardrails AI、NeMo Guardrails等）は日本語非対応だし、AI事業者GLのマッピングなんて当然ない。

結局、自分で作ることにした。それが [Aigis](https://github.com/killertcell428/aigis) なんだけど、この記事ではツールの宣伝ではなく、**37要件を1つずつ読み解いて、エンジニアとして何をすればいいのか** をまとめる。

同じ状況の人の参考になれば。

## v1.2 で何が変わったか

v1.1（25要件）→ v1.2（37要件）。追加12要件の中身はこう：

| 新規カテゴリ | 要件数 | ひとことで言うと |
|-------------|:------:|----------------|
| AIエージェント管理 | 2 | Claude Code等を「管理対象」として扱え |
| Human-in-the-Loop強化 | 3 | 緊急停止・最小権限・継続監視を義務化 |
| 新リスクカテゴリ | 4 | ハルシネーション・ディープフェイク・感情操作・AI依存 |
| 開発者責任 | 2 | RAG作った人も責任を負う |
| ガバナンス | 2 | 「攻めのガバナンス」（原文ママ） |

正直、「攻めのガバナンス」という表現は好き。守りだけじゃなく、ちゃんとAIを活用しろ、という趣旨。

## 37要件を分類してみた

全部並べると長いので、エンジニアが技術的に対応すべきものを中心に整理した。

### セキュリティ系（3要件）— ここが一番重い

**GL-SEC-01: 設計段階からのセキュリティ組み込み**

要はLLMの入出力にスキャン層を入れろ、という話。プロンプトインジェクション対策の実装が求められている。

```python
# 最低限これだけでも入れておくべき
from aigis import Guard

guard = Guard()
result = guard.check_input(user_message)
if result.blocked:
    return "不正なリクエストです"
```

何もしないで本番運用しているのが一番まずい。

**GL-SEC-03（v1.2新規）: 攻撃対象面の管理**

これがv1.2で一番インパクトのある要件だと思う。**MCPツールも攻撃対象面として管理しろ** ということ。

MCPサーバーの43%にコマンドインジェクション脆弱性があるという報告もあって、ツール定義をスキャンする仕組みが必要になった。実はこれに対応できるOSSは今のところほとんどない。

### Human-in-the-Loop系（4要件）— 地味に大変

**GL-HUMAN-02（v1.2新規）: 緊急停止メカニズム**

AIがCRITICALなリスクの操作をしようとしたとき、即座にブロックする仕組み。Slack通知でリアルタイムアラートを飛ばすのが現実的。

**GL-HUMAN-03（v1.2新規）: 最小権限の原則**

AIエージェントにsudoやrm -rfの実行権限を与えるな、ということ。当たり前に聞こえるけど、Claude Codeをデフォルト設定で使っていると、結構何でもできてしまう。

ポリシーエンジンで操作ごとにallow/deny/reviewを設定するのが正攻法。

### 新リスクカテゴリ（4要件）— v1.2の目玉

**GL-RISK-03: ハルシネーション起因の誤動作**

AIが「確認なし」で破壊的操作を実行するリスク。例えば「このファイルを削除して」とAIが勝手に解釈して実行する、みたいなケース。

**GL-RISK-04: 合成コンテンツ・フェイク情報**

ディープフェイク生成要求の検知。「この人の声で音声を作って」みたいなリクエストをブロックする。

**GL-RISK-06: 感情操作の防止**

これは面白い要件で、ダークパターンや心理操作をAIにやらせることを禁止している。「ユーザーの不安を煽って購入させろ」みたいな指示を検知する必要がある。

### AIエージェント管理（2要件）— v1.2の核心

**GL-AGENT-02: マルチエージェント連携の安全設計**

複数のAIエージェントが連携するとき、低権限のエージェントが高権限のエージェントに「セキュリティチェックを無効にしろ」と指示する攻撃（Second-Order Injection）がある。

エージェント間の委任関係を追跡し、権限昇格を防ぐ仕組みが必要。

### データ汚染対策（1要件）— RAGが対象

**GL-POISON-01: プロンプトインジェクション対策**

RAGで取得した外部ドキュメントに攻撃コードが含まれているケースへの対策。多層防御（正規表現 + 類似度検知 + 人間レビュー）が推奨されている。

```python
from aigis import scan_rag_context

chunks = retriever.search("user query")
result = scan_rag_context([c.text for c in chunks])
if not result.is_safe:
    # 汚染されたチャンクを除外
    chunks = [c for c in chunks if ...]
```

## 残りの要件（さらっと）

全部書くと長すぎるので、残りは箇条書きで：

- **GL-RISK-01/02**: リスクスコアリング（0-100）とインシデントDB。監査ログは必須
- **GL-TRANS-01/02**: 変更履歴の記録と能力・限界の開示
- **GL-DATA-01/02**: PII検知と自動墨消し。マイナンバー・クレカ等
- **GL-AUDIT-01**: 全操作の不変ログ保存
- **GL-RESP-01/02**: RAG構築者の責任。システムプロンプト漏洩対策
- **GL-GOV-01/02**: 段階的にガバナンスを強化する計画を立てろ

## やってみた感想

37要件を全部読んで対応してみて思ったこと：

1. **v1.2は「AIエージェント時代」を明確に意識している**。Claude CodeやCursorを使っている企業は、もう無視できない
2. **技術的に対応できる要件が大半**。ポリシーエンジン + 検知パターン + 監査ログで8割はカバーできる
3. **一番の壁は「何をすればいいかわからない」こと**。要件自体は合理的なので、一度理解すれば対応は難しくない

## コンプライアンスレポートを自動で出す

最終的に、37要件への対応状況をレポートとして出せるようにした。

```bash
aig report
# → AI事業者ガイドライン v1.2    37/37 covered (100.0%)
```

稟議書に「対応済み」と書くとき、根拠としてこのレポートを添付できる。地味だけど、これが一番実務で役に立つ機能だと思う。

## おわりに

AI事業者ガイドライン v1.2、最初は「面倒だな」と思ったけど、読めば読むほどまともなことが書いてある。特にv1.2で追加されたMCPの攻撃対象面管理や、マルチエージェントの安全設計は、実際にインシデントが起きている領域なので、対応しておいて損はない。

自分が作ったAigisは [GitHub](https://github.com/killertcell428/aigis) で公開している。37要件のマッピングは `aig report` で確認できるので、同じ悩みを持つ人は試してみてほしい。

ゼロ依存（Python標準ライブラリのみ）なので、`pip install aigis` だけで動く。
