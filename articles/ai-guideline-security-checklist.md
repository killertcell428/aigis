## AI事業者ガイドライン第1.1版とは

2025年3月28日、総務省と経済産業省は「AI事業者ガイドライン」の第1.1版を公表しました。

> AI事業者ガイドライン（第1.1版）
> 令和7年3月28日 総務省 経済産業省
> — [総務省 公式掲載ページ](https://www.soumu.go.jp/main_sosiki/kenkyu/ai_network/02ryutsu20_04000019.html)

このガイドラインは「Living Document」として位置づけられ、AIをめぐる急速な技術進歩・国際動向に合わせて継続的に更新されています。第1.0版（2024年4月）からの主な改定ポイントは以下の7つです。

1. **AIリスクの洗い出し・分類の強化**
2. **AI契約に関する留意事項の追加**
3. **生成AI（マルチモーダル・RAG・コード生成）の具体的記載**
4. **AIガバナンス事例の充実**（リスクベースアプローチ、人材不足対応）
5. **国際動向の反映**（広島AIプロセス、EU AI法）
6. **重要用語（バイアス・透明性等）の定義整理**
7. **AIエージェント・フィジカルAIへの言及**

> 出典: [ニュートンコンサルティング — 生成AIに関する記載を拡充、AI事業者ガイドラインの第1.1版](https://www.newton-consulting.co.jp/itilnavi/flash/id=7978)

特に注目すべきは、**AIエージェント**と**フィジカルAI**に関する記述が新たに追加されたことです。ガイドラインでは、AIエージェントの登場により「事故等の安全性面のリスクや過度な依存、労働者の失業等のリスクが複雑化・深刻化する可能性がある」と明記されています。

> 出典: [AISI（AIセーフティ・インスティテュート）— AI事業者ガイドライン(1.1版)が公表されました](https://aisi.go.jp/output/output_information/250328_2/)

## ガイドラインの10原則 — AIエージェント視点で読み解く

ガイドラインは全主体（開発者・提供者・利用者）に共通する**10の基本原則**を定めています。ここでは、AIエージェント環境で特に重要な原則を深掘りします。

> 出典: [経産省「AI事業者ガイドライン v1.1」対応チェックリスト — リスキリングナビ](https://reskilling-navi.com/column/meti-ai-guideline-v11-checklist)

### 原則5: セキュリティ確保

> 「脅威対策と脆弱性管理」

AIエージェントが社内システムにアクセスし、ファイル操作やコマンド実行を行う環境では、従来のWebアプリケーションとは異なるセキュリティリスクが存在します。

**AIエージェント特有のリスク:**
- **プロンプトインジェクション**: 悪意ある入力でエージェントの動作を操作される
- **ツール呼び出しの悪用**: エージェントが意図しないコマンドを実行する
- **認証情報の漏洩**: 環境変数やAPIキーがエージェント経由で外部に送信される
- **サプライチェーン攻撃**: 依存パッケージ経由でのマルウェア混入（[litellm事件](https://docs.litellm.ai/blog/security-update-march-2026)が典型例）

### 原則6: 透明性向上

> 「判断プロセスや設計情報の可視化」

AIエージェントは自律的に判断・行動するため、**何をしたのか・なぜそうしたのかを事後的に追跡できること**がガバナンス上不可欠です。

**具体的に求められるもの:**
- 意思決定プロセスのログ記録と開示
- 利用データやアルゴリズムの概要説明
- ユーザーへのフィードバック機能の提供

### 原則7: 説明責任

> 「責任の明確化と記録管理」

トラブル発生時に「AIがやりました」では済みません。以下が必須です。

- トラブル時の責任所在の明確化
- **人間の最終責任者の明示**
- **意思決定プロセスの記録と監査可能性の確保**
- 自動実行の範囲と人手による対応の境界の定義

> 出典: [GVA法律事務所 — AI事業者ガイドライン改訂の要点](https://gvalaw.jp/blog/i20260303/)

## 主体別チェックリスト — AIエージェント導入企業の実務

ガイドラインは「AI開発者」「AI提供者」「AI利用者」の3つの主体に対してそれぞれ遵守事項を定めています。AIエージェントを社内導入する企業は、多くの場合**利用者**の立場に該当しますが、エージェントの設定やカスタマイズを行う場合は**提供者**としての責務も負います。

### AI利用者向けチェックリスト

| # | チェック項目 | ガイドライン原則 | 対応レベル |
|---|-----------|--------------|----------|
| 1 | AIの出力結果を過信せず、人間の最終判断を介在させる体制があるか | 人間中心 | 必須 |
| 2 | AIエージェントの操作を記録し、事後的に監査できるか | 透明性・説明責任 | 必須 |
| 3 | 個人情報や機密データのAIへの入力を制限する仕組みがあるか | プライバシー | 必須 |
| 4 | AIエージェントが実行できる操作の範囲を制限しているか | 安全性 | 必須 |
| 5 | 危険な操作（ファイル削除、外部通信等）をブロックする機構があるか | セキュリティ | 強く推奨 |
| 6 | AIの判断プロセスをユーザーに説明できるか | 透明性 | 推奨 |
| 7 | インシデント発生時の責任者・連絡先が明確か | 説明責任 | 必須 |
| 8 | AI利用に関する社内教育・リテラシー向上を行っているか | 教育 | 推奨 |

### AI提供者向け追加チェック

| # | チェック項目 | ガイドライン原則 |
|---|-----------|--------------|
| 9 | AIの性能範囲・限界・想定利用シーンを明示しているか | 透明性 |
| 10 | 継続的モニタリングとフィードバックの仕組みを構築しているか | 安全性 |
| 11 | バイアス・セキュリティリスクへの対応体制を整備しているか | 公平性・セキュリティ |
| 12 | 契約書やポリシーで責任範囲を明確化しているか | 説明責任 |

### AI開発者向け追加チェック

| # | チェック項目 | ガイドライン原則 |
|---|-----------|--------------|
| 13 | 認証・暗号化・ログ管理を含むセキュリティ設計がされているか | セキュリティ |
| 14 | バイアスや不当な差別が入り込まない設計・評価を行っているか | 公平性 |
| 15 | モデル仕様・学習内容・評価結果を記録・文書化しているか | 透明性 |

> 出典: [経産省「AI事業者ガイドライン v1.1」対応チェックリスト — リスキリングナビ](https://reskilling-navi.com/column/meti-ai-guideline-v11-checklist)

## なぜ「監査ログ」と「操作制御」が最重要なのか

チェックリストの中で、AIエージェント環境で最もクリティカルなのは以下の2点です。

**1. 操作ログの記録と監査可能性（チェック#2）**

ガイドラインが求める「透明性」と「説明責任」を同時に満たすには、エージェントの全操作を記録し、いつでも監査できる状態にしておく必要があります。

- 「いつ・誰が・どのAIに・何を指示し・何が実行されたか」の5W1H
- 危険操作がブロックされた場合のログ
- 操作の成功/失敗と影響範囲

**2. 危険操作のブロック機構（チェック#5）**

AIエージェントが`rm -rf /`や`curl | sh`を実行しようとしたとき、事前に検知・ブロックする仕組みが必要です。これは「安全性確保」と「セキュリティ確保」の両原則に直接対応します。

## 新年度を迎えるにあたって — 企業のAIガバナンス実装のタイミング

AI事業者ガイドライン第1.1版の公表タイミングは、2025年度の開始（4月）と重なっています。多くの企業が新年度のAI導入計画を立てるこのタイミングで、ガバナンス体制を整備しておくことは極めて重要です。

Gartnerのレポートによれば：

- Fortune 500企業の**80%**がAIエージェントを導入済み（2026年時点）
- そのうち**63%**がガバナンス課題に直面
- 2027年までにAIプロジェクトの**40%**がガバナンス不足で失敗すると予測

> 出典: [Gartner — セキュリティ対策が不十分なAIエージェントはサイバー攻撃の温床](https://www.gartner.co.jp/ja/newsroom/press-releases/pr-20260316-ai-agent-sec)

つまり、**ガイドラインへの対応はコンプライアンスのためだけでなく、プロジェクトの成功確率を上げるための実務的な投資**です。

## aigisでガイドライン対応を実装する

ここまで見てきたチェック項目のうち、技術的に対応が必要な部分を`pip install`一発で導入できるOSSがあります。

[**aigis**](https://github.com/killertcell428/aigis)は、AIエージェントの操作を監視・制御するオープンソースのLLMセキュリティライブラリです。

```bash
pip install aigis
aig init
```

### チェックリストとaigisの対応表

| チェック項目 | aigisの対応機能 |
|-----------|-------------------|
| #2 操作ログの記録と監査 | **Activity Stream**: 全ツール呼び出しをJSON Lines形式で自動記録。タイムスタンプ・操作内容・判定結果を含む |
| #3 機密データの入力制限 | **PII検出スキャン**: マイナンバー・クレジットカード番号・メールアドレス等の個人情報を検出しブロック |
| #4 操作範囲の制限 | **ポリシーエンジン**: YAML形式でallow/deny/reviewルールを定義。ファイルパス・コマンド・ネットワークアクセスを制御 |
| #5 危険操作のブロック | **脅威スキャン（48パターン）**: プロンプトインジェクション、ジェイルブレイク、危険なコマンド実行を検知・ブロック |
| #7 インシデント対応 | **監査レポート生成**: 操作ログから自動的にレポートを生成し、インシデント調査を支援 |
| #13 セキュリティ設計 | **Claude Code Hook**: PreToolUseフックでツール実行前に検査。exit 0で許可、exit 2でブロック |

### ポリシー定義の例

```yaml
# policy.yaml — 社内ガイドライン準拠ポリシーの例
version: "1.0"
rules:
  - name: "機密ファイルへのアクセス制限"
    action: deny
    conditions:
      file_path:
        - "/etc/passwd"
        - "**/.env"
        - "**/credentials*"
    message: "機密ファイルへのアクセスはポリシーにより禁止されています"

  - name: "外部通信の制御"
    action: review
    conditions:
      command:
        - "curl*"
        - "wget*"
    message: "外部通信は管理者のレビューが必要です"

  - name: "危険なコマンドのブロック"
    action: deny
    conditions:
      command:
        - "rm -rf*"
        - "chmod 777*"
        - "* | sh"
        - "* | bash"
    message: "危険なコマンドの実行はブロックされました"
```

## まとめ

AI事業者ガイドライン第1.1版は、AIエージェント時代の到来を見据えた重要な改定です。特に企業がAIエージェントを導入する際に求められるのは、以下の3つです。

1. **操作ログの記録と監査可能性の確保**（透明性・説明責任）
2. **危険操作の事前検知とブロック**（安全性・セキュリティ）
3. **機密データの保護と入力制限**（プライバシー）

これらは単にコンプライアンス対応のためではなく、AIプロジェクトを成功させるための実務的な基盤です。新年度のAI導入計画を立てる際に、ガバナンス体制を組み込んでおくことをお勧めします。

aigisは `pip install aigis` でこれらの要件を技術的にカバーするOSSです。導入ガイドの詳細は[こちらの記事](https://zenn.dev/sharu389no/articles/e07c926d87ac57)もご参照ください。
