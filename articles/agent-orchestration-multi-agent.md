## はじめに

「AIに商品企画を丸ごと任せたらどうなるか？」

この問いに本気で取り組んでみた。1つのテーマを投げるだけで、市場調査、顧客分析、競合分析、コンセプト策定、PRD作成、ネーミング開発、デザインレビュー、プレゼン資料生成まで——商品企画の全工程を自律的に完走する「AI管理職」を Claude Code のスキルシステムで構築した話だ。

人間（CEO役）が判断するのは、たった4回。それ以外はすべてAIが自律判断する。

結果として、LED法人向け新商品「KANTERI（カンテリ）」の企画を1日で完走し、Excel 7シート・Word 6本・PowerPoint・画像4枚の計15成果物を生成した。TAM 5,580億円、3年ROI 74.5倍の事業計画付きで。

この記事では、そのアーキテクチャと実行フローを、実データとともに解説する。

## アーキテクチャ — product-planning-orchestrator の設計思想

### 「AI管理職」という比喩が意味するもの

普通のAIツールは「指示されたことをやる」。AIアシスタントだ。このオーケストレーターは違う。**自分で判断し、自分で次のアクションを決め、品質が基準に達するまで自分でやり直す**。人間の管理職がやっていることと同じだ。

設計の核心はこの3つ：

1. **CEOへの確認は4回だけ** — キックオフ・コンセプト承認・商品名決定・最終レビュー
2. **10以上のサブスキルをファイルベースで疎結合連携** — 各スキルが独立しつつ、成果物を通じて情報を受け渡す
3. **品質ゲートコントローラー（QGC）で自律的に品質を維持** — 基準未満なら自動で修正を繰り返す

### Claude Code のスキルシステム

Claude Code には `.claude/skills/` にスキル定義（SKILL.md）を置くと、自然言語の指示でスキルを呼び出せる仕組みがある。今回のオーケストレーターは、以下のサブスキルを順番に呼び出して企画を進める。

```
product-planning-orchestrator（AI管理職）
├── market-research          # PESTLE・Five Forces・市場規模
├── customer-insight         # JTBD/ODI・ペルソナ・購買行動
├── competitive-analysis     # 競合4社比較・ウォーゲーミング
├── concept-sheet            # S.R.G判定コンセプトシート
├── prd-create               # 5W3H・11要件のPRD
├── product-naming           # 20候補→7軸評価→TOP3
├── design-review            # 5部門14指摘のレビュー
├── slide-sheet-create       # Excel/PPTX生成
├── research-web             # Web検索（各スキルが内部利用）
└── kickoff-checklist        # 5部門32タスクのチェックリスト
```

### データハブ：`_AI_knowledge/`

すべてのスキルが成果物を出力し、次のスキルが読み込む場所が `_AI_knowledge/` フォルダだ。

## 実行フロー詳細 — KANTERI（LED法人向け新商品）を例に

実際にオーケストレーターを動かした「KANTERI_2026」プロジェクトの実行フローを追う。

### Phase 0: テーマ投入 → AI仮説生成 → CEO方向性確認

ユーザー（CEO役）がテーマを投げる。

> 「LEDの商品企画を一通りアップデートしよう！アイリスの商品企画担当者を驚かす完成度の高さの成果物を作りたい！」

AIはこれだけの情報から仮説を立てる：

- **ターゲット**: 従業員50-300名の中小企業
- **価格帯**: 8,000-15,000円/台
- **差別化6点**: 工事不要 / Bluetooth Mesh / 5分アプリセットアップ / 明朗会計 / 補助金サポート / ウェルビーイング照明

CEOのフィードバック：

> 「差別化を作ろうとしすぎて、のちに開発や製造部分でコストがオーバーしそう。核となる機能にフォーカスしたプランがいい」

AIが修正——差別化を6点から3点に絞り込む：

1. **工事不要スナップオン設置**（製造コスト増なし）
2. **パッケージ明朗会計**（仕組みの差別化、コスト増ゼロ）
3. **補助金申請まるごとサポート**（サービスの差別化、コスト増ゼロ）

「機能で勝つ」のではなく「体験と仕組みで勝つ」——この方向性が確定した瞬間だ。

### Phase 1 (Gate 1): 市場調査 → 顧客インサイト → 競合分析 → コンセプト

ここからはAIが自律実行する。CEOに確認するのはGate末尾の1回だけ。

**STEP 1-1: 市場調査（PESTLE / Five Forces）**

Web検索10テーマ・30件以上のソースを元に定量分析を実行。

- 市場全体: 4,862億円（2025年LED器具市場、富士経済）
- 対象市場: 1,280億円（中小企業向け）
- 獲得目標: 128-192億円（ベースシナリオ）
- 蛍光灯置換特需: 年400-600億円（2026-2028年）

**STEP 1-2: 顧客インサイト（JTBD / ODI / ペルソナ3名）**

実際のレビュー・口コミデータから顧客ニーズを抽出。

- 最も満たされていないニーズ: 「総費用の事前把握」
- 2位: 「工事なしでの交換完了」
- 過剰充足（外すべき機能）: IoT、調光、人感センサー
- 購買プロセスの最大離脱点: **見積もり段階（約60%が離脱）**

**STEP 1-3: 競合分析（4社比較）**

- 直接競合3者: 大塚商会、東芝TENQOO、中国低価格ブランド
- 中小企業が重視する4観点の加重評価: KANTERI **4.65** > 大塚商会 3.22 > 東芝 2.78
- 8,000-15,000円 x 工事不要 x 明朗価格 x 補助金サポートの組み合わせは**競合不在**
- 最大の脅威は競合ではなく「何もしない（先延ばし行動）」

**STEP 1-4: コンセプト承認（S.R.G判定）**

| 判定 | 結果 | 根拠 |
|------|------|------|
| **S（Simple）** | ◎ | 機能を「照らす・取り付ける・保証する」の3つに集中 |
| **R（Reasonable）** | ◎ | 8,000-15,000円（工事費ゼロで実質最安クラス）、補助金適用で回収11ヶ月 |
| **G（Good）** | ◎ | 最も満たされていない顧客ニーズを直接解決、既存資産を最大活用 |

→ **CEO判断: GO**

### Phase 2 (Gate 2): PRD → ネーミング → デザインレビュー

**STEP 2-1: PRD（5W3H / 11要件）**

- 2 SKU: ベースライト型 LCX-40B / 直管型 LCX-40T
- 必須機能5つ: スナップオン取付 / 明朗会計 / 補助金テンプレート / 170lm/W / 3年保証
- 原価率: 33.6%、粗利率: 60%以上
- 4ティアパッケージ価格（1台/20台/50台/100台）
- 投資: 金型1,100万円 + 営業ツール・販促4,000万円 = 合計5,100万円

**STEP 2-2: ネーミング（20候補 → 7軸評価 → TOP3）**

v01で「LiCONEX」を提案 → 不適合（"Connect"がコスト削減方針とズレ）→ v02で35候補を再生成。

- 1位: **KANTERI（カンテリ）** — 103/120点
- 2位: RAKUTERI — 96/120点
- 3位: PACHITTO — 96/120点

→ **CEO判断: 1位 KANTERI を採用**

**STEP 2-3: デザインレビュー（5部門14指摘）**

| 部門 | 判定 | 主要指摘 |
|------|------|---------|
| 開発部 | 条件付き承認 | スナップオン15パターン適合テスト未実施 |
| 品質管理部 | 承認 | 3年保証コスト定量化・MTBF目標設定が必要 |
| 製造部 | 承認 | 旧正月前の生産タイムライン+金型投資額明記 |
| 営業部 | 条件付き承認 | 商品特長説明研修モジュール作成が必要 |
| 知財部 | 条件付き承認 | KANTERI商標Class 11出願を2026年4月中に |

### Phase 3 (Gate 3): プレゼン資料 → 成果物パッケージ

**プレゼン会議資料（Excel 7シート）**

アイリスオーヤマのプレゼン会議はPowerPointではなくExcelで行う。これが面白いところだ。1つのExcelファイルに全情報を集約し、発表シートと質問対応シートを分ける。

| シート | 内容 | 行数 |
|--------|------|------|
| 01_サマリ | 決裁事項・S.R.G判定・価格比較・市場動向・製品画像 | 45行 |
| 02_市場データ | サマリー + 市場調査レポート全文 | 518行 |
| 03_企画商品概要 | サマリー + PRD全文 | 452行 |
| 04_競合比較 | サマリー + 競合分析レポート全文 | 425行 |
| 05_顧客インサイト | サマリー + 顧客インサイトレポート全文 | 514行 |
| 07_想定Q&A | 9カテゴリ51問（ステータス色分け） | 80行 |
| 09_リスク・課題 | 8リスク（確率x影響度+対策） | 11行 |

各シートが「上部=結論サマリー → 区切り線 → 下部=分析の全詳細」のブレイクダウン構造。社長の「根拠は？」に対して、同じシート内でスクロールするだけで全データを見せられる。

→ **CEO判断: 最終レビュー承認**

## エージェント間データ受け渡し — ファイルベースの疎結合通信

マルチエージェントシステムでよく議論される「エージェント間通信」をどう実装したか。

### なぜファイルベースなのか

API呼び出しやメッセージキューではなく、**Markdownファイルへの読み書き**でエージェント間を連携させている。理由は3つ：

1. **人間が読める** — 中間成果物がすべてMarkdownなので、いつでも人間が確認・修正できる
2. **デバッグしやすい** — どのステップで何が生成されたか、ファイルを見れば一目瞭然
3. **Claude Code との相性** — Claude Code はファイル読み書きが最も得意な操作

### データフロー

```
テーマ投入
  ↓
[market-research] → 20260317_市場調査レポート_v02.md
  ↓ (読み込み)
[customer-insight] → 20260317_顧客インサイトレポート_v02.md
  ↓ (読み込み)
[competitive-analysis] → 20260317_競合分析レポート_v02.md
  ↓ (全3ファイル読み込み)
[concept-sheet] → S.R.G判定 → CEO確認
  ↓
[prd-create] → 20260317_製品要件定義書_v02.md
  ↓ (読み込み)
[product-naming] → 20260317_ネーミング提案_v02.md → CEO確認
  ↓
[design-review] → 20260317_デザインレビュー_v02.md
  ↓ (全ファイル統合)
[slide-sheet-create] → presentation_data.json → Excel/PPTX
  ↓
project_summary.json（最終メタデータ）
```

各スキルは前のステップのMarkdownを読み込んで参照する。例えば `customer-insight` は市場調査レポートの市場規模データを引用し、`competitive-analysis` は顧客インサイトの「過剰充足スコア」を使って「外すべき機能」を特定する。

### バージョン管理と矛盾解決

- `v01` → QGCで品質不足 → `v02` を生成（`v01` は残す）
- 下流スキルは常に最新バージョンを参照する
- 上流の修正が下流と矛盾する場合、QGCが検出してフラグを立てる

## 品質管理の仕組み — QGC（Quality Gate Controller）

「AIが作った企画なんて、所詮それっぽいだけでしょ？」

この批判に真正面から応えるのがQGCだ。

### 評価の仕組み

各ステップ完了時に、Critic（批評者）ロールでセルフレビューを実施する。

```
評価軸の例（市場調査の場合）:
- C1: 論理構造（主張→根拠→示唆の一貫性）
- C2: データの信頼性（出典URL付き・三角測量）
- M1: TAM/SAM/SOMの計算根拠
- M2: 競合環境の定量評価
- M3: 自社への示唆の具体性

各軸 5段階評価:
- 5: 社長プレゼンに耐える品質
- 4: 修正なしで次のステップに渡せる
- 3: 部分的に修正が必要
- 2: 大幅な修正が必要
- 1: やり直し
```

### 自律修正ループ

```
生成（v01） → QGC評価
                ↓
          全軸 ≥ 4/5 → PASS → 次のステップへ
                ↓
          いずれか < 4 → 改善指示書を自動生成
                ↓
          改善実行（v02） → QGC再評価
                ↓
          ... 最大3イテレーション ...
                ↓
          3回修正してもNG → CEOにエスカレーション
```

実際のKANTERIプロジェクトでは、ほぼすべてのレポートが `v01` → `v02` のイテレーションを経ている。`v01` でのネーミング「LiCONEX」が方針とズレていることをQGCが検出し、`v02` で35候補の再生成に至ったケースが典型的だ。

### 最終品質スコア

最終レビュー時の品質評価は3カテゴリ37点満点：

| カテゴリ | 内訳 | 配点 |
|---------|------|------|
| **Logic（論理構造）** | 主張-根拠の一貫性、矛盾検出 | 6点 |
| **Content（コンテンツ）** | データ精度、分析深度、実行可能性 | 25点 |
| **Iris適合性** | アイリスの企業文化・プレゼン様式への適合 | 6点 |

「社長の『で、いくら儲かるの？』『根拠は？』に耐えられるか」が品質の基準。抽象的な表現や根拠のない楽観はQGCが弾く。

- **`product_name_prev`** — QGCによる名称変更の履歴が残っている

プレゼン会議で想定される質問を、役職・部門別に51問用意した。

| 開発部 | 8問 | 回答可2、テスト未実施3、要確認2、設計段階1 |

## 学びと課題

### うまくいったこと

**1. CEOタッチポイントの最小化**

4回の確認で、方向性の軌道修正が十分にできた。特にPhase 0でのCEOフィードバック（差別化6点→3点への絞り込み）は、この後のすべてのステップの品質を決定づけた。人間の判断は「量」ではなく「質」が重要だという当たり前のことをシステムとして実装できた。

**2. 品質ゲートによる自律的品質維持**

ネーミングの「LiCONEX → KANTERI」への変更がわかりやすい例だ。v01の名称が「Connect=つながる」を含意していたが、コスト削減方針（IoTや無線制御を外す方向）と矛盾していることをQGCが検出した。人間なら「なんかしっくりこないな」と感じるレベルの違和感を、AIが論理的に言語化して修正プロセスに乗せた。

**3. 引き算の企画ができた**

AIは「足す」のが得意で「引く」のが苦手、というのが一般的な印象だろう。しかし、ODI（Outcome-Driven Innovation）のフレームワークをスキルに組み込むことで、過剰充足スコアが高い機能（人感センサー: 4.5、調光調色: 4.0、Bluetooth: 3.5）を定量的に特定し、**意図的に外す判断**ができた。

### 課題

**1. Web検索の精度限界**

市場規模データの三角測量（複数ソースからの検証）を仕組みとして入れているが、それでもWeb検索で取得できるデータには限界がある。有料データベース（富士経済、矢野経済など）のフルレポートにはアクセスできないため、公開データの範囲での推計になる。

**2. 画像生成の品質**

製品イメージ4枚（product / lifestyle / detail / feature_callout）を生成したが、実際の製品デザインとしてはまだ「イメージ」の域を出ない。カタログに載せるレベルではなく、あくまで「方向性の共有」用途だ。

**3. リアルタイム市場データの取得**

2027年の蛍光灯製造禁止に向けた市場動向など、時々刻々と変わるデータを最新の状態で取得し続けることが難しい。定期的な再調査の仕組みが必要になる。

### 次のステップ

**並列実行** — 現在は直列実行だが、市場調査と競合分析は並列実行できるはず。Claude Code の SubAgent を活用すれば、Phase 1 の実行時間を半分にできる可能性がある。

**外部API連携** — 特許データベース（J-PlatPat）、企業情報（帝国データバンク）、市場データ（Statista）などとのAPI連携。現在のWeb検索ベースの情報収集を、より信頼性の高いデータソースに置き換える。

**フィードバックループ** — 複数プロジェクトの `project_summary.json` を横断分析し、「どの市場の企画が成功しやすいか」「どのフレームワークが品質スコアに寄与しているか」をメタ分析する仕組み。すでに `/project-compare` スキルで横断比較Excelを生成できるが、これを学習に活かすステップがまだない。

## おわりに

「AI管理職」は、AIが人間の仕事を奪う話ではない。**人間が本当に判断すべき4つのポイントに集中できるようにする仕組み**だ。

商品企画の仕事の大部分は、調査、分析、文書化、整合性チェックといった「知的だが定型的な」作業で構成されている。そこをAIに任せて、人間は「この方向性でいいか」「この名前でいいか」「この企画を世に出すか」という本質的な判断に集中する。

Claude Code のスキルシステムとファイルベースの疎結合アーキテクチャは、この「AI管理職」を実現する上で想像以上に柔軟だった。スキルの追加・修正がMarkdownファイルの編集だけで完結し、エージェント間のデータ受け渡しが人間にも読めるMarkdownで行われる。ブラックボックスにならない。

次は、このオーケストレーターをさらに磨いて、他の業務領域——例えば営業戦略立案やM&Aデューデリジェンス——に横展開できるか、実験してみたいと思っている。

テーマを1つ投げるだけで、1日後には15成果物の企画パッケージが手元にある。これが2026年の商品企画の姿だ。
