# ガバナンス体制の構築

## なぜAIエージェント専用のガバナンスが必要なのか

AIエージェントを本番導入する組織が急増する一方で、ガバナンス体制の整備は大きく遅れています。Gartnerの調査によれば、AIガバナンスが成熟していると評価できる企業は全体の約20〜25%にとどまり、AIエージェント固有の管理戦略を持つ企業はわずか約10%です。

McKinseyの2024年調査では、AI採用組織の72%が何らかのガバナンスポリシーを保有していると報告していますが、その多くは従来型のMLモデル向けに設計されたものであり、エージェント特有のリスクに対応できていません。

エージェント固有のガバナンスが必要な理由は、従来のAIガバナンスがカバーしない領域が存在するからです。

```
従来のAIガバナンス:
  - モデルの精度・公平性・説明可能性
  - データプライバシー・バイアス管理
  - 入出力の品質管理

エージェント固有のガバナンス（追加領域）:
  - ツール呼び出しの許可・制限ポリシー
  - 自律的意思決定の境界設定
  - マルチエージェント間の権限委任ルール
  - Human-in-the-Loopの発動条件
  - 外部サービス連携のセキュリティ要件
```

従来のガバナンスが「モデルの出力品質」に焦点を当てていたのに対し、エージェントガバナンスは「行動の範囲と権限」を制御する必要があります。エージェントはファイル操作、API呼び出し、データベース変更といった不可逆なアクションを実行するため、事後の修正が困難です。したがって、事前に行動の境界を定義し、リアルタイムで監視するガバナンス体制が不可欠なのです。

## ガバナンス成熟度モデル

組織のエージェントガバナンス成熟度は、以下の5段階で評価できます。自社がどの段階にいるかを把握することが、適切な投資判断の出発点になります。

### Level 1: Ad Hoc（場当たり的）

ガバナンスポリシーが存在しないか、個人の判断に依存している段階です。各チームが独自にエージェントを導入し、セキュリティ要件もまちまちです。多くの組織がこの段階にあり、インシデント発生時に初めて問題が顕在化します。

### Level 2: Reactive（事後対応型）

インシデントをきっかけにポリシーが策定され始める段階です。利用ガイドラインは存在しますが、技術的な強制力を持ちません。「〜してはならない」という禁止事項リストがドキュメントとして共有される程度です。

### Level 3: Defined（定義済み）

組織横断的なポリシーが文書化され、責任分担が明確になっている段階です。RACIマトリクスが定義され、エージェント導入の承認プロセスが標準化されています。ただし、ポリシーの適用は手動で行われ、網羅性に課題があります。

### Level 4: Managed（管理型）

ポリシーがコードとして実装され、自動的に適用される段階です。ガードレール違反がリアルタイムで検出・ブロックされ、監査証跡が自動生成されます。ガバナンスのPDCAサイクルが回り始めています。

### Level 5: Optimized（最適化型）

ガバナンスメトリクスに基づく継続的改善が実現している段階です。リスクベースのポリシー更新、レッドチーミングの定期実施、規制変更への迅速な対応が組織的に行われています。

大半の組織はLevel 1〜2にとどまっています。本章では、Level 3〜4への移行に必要な具体的ステップを解説します。

## ポリシー as Code

エージェントガバナンスを実効性あるものにするためには、ポリシーを自然言語のドキュメントではなく、機械的に評価可能なコードとして定義する「ポリシー as Code」アプローチが不可欠です。

### なぜポリシーをコード化するのか

自然言語のポリシーには3つの本質的な弱点があります。第一に**曖昧性**です。「機密データを適切に扱うこと」という規定は、人によって解釈が異なります。第二に**適用の不完全性**です。すべてのエージェントの全呼び出しを人間がレビューすることは現実的ではありません。第三に**監査の困難性**です。ポリシー違反の有無を事後に検証する手段がありません。

ポリシー as Codeは、これらの問題をすべて解決します。ポリシーはコードとして記述されるため曖昧性がなく、自動評価によって全呼び出しに適用され、評価ログが監査証跡として残ります。

### OPA（Open Policy Agent）

OPA（Open Policy Agent）はCNCFのGraduatedプロジェクト（2021年2月卒業）であり、ポリシー as Codeの業界標準ツールです。Rego言語でポリシーを記述し、JSON入力に対して評価結果を返します。

```rego
# OPA Rego: エージェントのファイルアクセスポリシー例
package agent.file_access

default allow = false

allow {
    input.action == "read"
    startswith(input.path, "/data/public/")
}

allow {
    input.action == "read"
    input.path == "/data/reports/"
    input.agent.role == "analyst"
}

deny {
    input.action == "write"
    startswith(input.path, "/etc/")
}
```

OPAの強みは汎用性にあります。Kubernetes Admission Control、API Gateway、マイクロサービスの認可など、幅広い領域で採用実績があります。一方で、Rego言語の学習コストが導入障壁となるケースも少なくありません。

### AWS Cedar

AWS Cedarは2023年5月にオープンソースとして公開されたポリシーエンジンで、型付きポリシー言語、RBAC（Role-Based Access Control）とABAC（Attribute-Based Access Control）のネイティブサポート、**形式検証**（ポリシーの矛盾や到達不能ルールの自動検出）が特徴です。

```cedar
// Cedar: エージェントのAPI呼び出しポリシー例
permit(
    principal == Agent::"data-analyst",
    action == Action::"api_call",
    resource == API::"internal-analytics"
) when {
    context.time.hour >= 9 && context.time.hour <= 18
};

forbid(
    principal in AgentGroup::"external-facing",
    action == Action::"database_write",
    resource in Database::"production"
);
```

Cedarの形式検証は、ポリシーの品質保証において大きな価値を持ちます。ポリシー間の矛盾（同一条件でpermitとforbidが両立する）やシャドウルール（他のルールによって到達不能になるルール）を静的に検出できるため、本番適用前にポリシーのバグを発見できます。

### ポリシーエンジンの選定基準

OPAとCedarはそれぞれ異なる強みを持ちます。OPAは汎用性とエコシステムの成熟度で優れ、Cedarは型安全性と形式検証で優れます。選定の基準は、既存のインフラ（AWS中心ならCedar、Kubernetes中心ならOPA）とチームのスキルセットに依存します。

いずれのエンジンを選択する場合も、重要なのは**評価速度**です。エージェントのツール呼び出しは一つのタスクで数十〜数百回に達するため、ポリシー評価はサブミリ秒で完了する必要があります。OPAもCedarも決定論的な評価エンジンであり、この要件を満たします。LLMベースのポリシー評価は柔軟性がありますが、レイテンシとコストの観点からリアルタイム適用には不向きです。

## AI CoEの設置と運用

ポリシー as Codeの技術基盤が整っても、それを組織的に運用する体制がなければ機能しません。多くの先進企業では、AI Center of Excellence（AI CoE）を設置し、AIガバナンスの推進役としています。

### 先行事例

**JPMorgan Chase**は社内でLLM Suiteを開発・運用し、従業員の60%が利用するまでに成長させています。このスケールでのAI利用を支えているのが、全社横断のAI CoEです。ガバナンスポリシーの策定、ユースケースの審査、セキュリティレビューを一元的に管理しています。

**Shell**は1,000を超えるAIユースケースを運用しており、AI CoEがユースケースの分類、リスク評価、ガバナンスフレームワークの適用を統括しています。

**Walmart**も大規模なAI CoEを設置し、調達から運用まで一貫したAIガバナンスを実施しています。

### AI CoEの構成と役割

AI CoEは以下の4つの機能を持つべきです。

**ポリシー策定機能**: エージェントの利用ポリシー、セキュリティ要件、データ取り扱い規定を策定し、ポリシー as Codeとして実装します。

**審査・承認機能**: 新規エージェント導入のリスクアセスメント、ユースケースの審査、本番環境への昇格判断を行います。

**モニタリング機能**: 運用中のエージェントのガードレール違反、異常行動、パフォーマンス指標を監視し、是正措置を講じます。

**教育・啓発機能**: 開発者向けのセキュアエージェント開発ガイドライン、経営層向けのリスク報告、全社向けのAIリテラシー教育を提供します。

### RACIマトリクス

AI CoEの設置にあたり、関係部門の責任分担を明確化するRACIマトリクスの定義が重要です。

| 活動 | AI CoE | 開発チーム | セキュリティ | 法務 | 経営層 |
|---|---|---|---|---|---|
| ポリシー策定 | A/R | C | C | C | I |
| エージェント開発 | C | A/R | C | I | I |
| セキュリティレビュー | R | C | A/R | I | I |
| 本番昇格判断 | A | R | R | C | I |
| インシデント対応 | R | R | A/R | C | I |
| 規制対応 | R | I | C | A/R | I |

*R=Responsible, A=Accountable, C=Consulted, I=Informed*

小規模な組織ではAI CoEを独立した組織として設置することが難しい場合があります。その場合は、既存のセキュリティチームやプラットフォームチームに機能を内包する「バーチャルCoE」モデルが現実的です。重要なのは、ガバナンスの責任者が明確に定義されていることです。

## ツール紹介

### Aigis — YAMLポリシーエンジン

Aigisは、YAMLベースのポリシー定義によるエージェントガバナンスを提供します。14のビルトインルールが組み込まれており、すぐに利用を開始できます。

```yaml
# aigis ポリシー定義例
rules:
  - id: block-sensitive-file-access
    description: "機密ファイルへのアクセスをブロック"
    action: deny
    conditions:
      - tool: file_read
        path_pattern: "/etc/shadow|/etc/passwd|*.pem|*.key"
  - id: require-review-for-api-calls
    description: "外部API呼び出しは人間レビュー必須"
    action: review
    conditions:
      - tool: api_call
        destination: "external"
```

業種別テンプレートが7種類（finance、healthcare、ecommerce、internal、education、customer_support、developer_tools）用意されており、自社の業種に合わせたポリシーを短時間でセットアップできます。たとえば金融業界テンプレートでは、PII（個人識別情報）の外部送信ブロック、取引データへのアクセス制御、監査ログの詳細記録がデフォルトで有効になっています。

Aigisのポリシー評価は決定論的（正規表現ベース）であり、LLMを経由しません。このため評価速度はサブミリ秒であり、エージェントのツール呼び出し頻度が高い環境でもボトルネックになりません。

### OPA / AWS Cedar

OPAとCedarは、前述のとおりポリシー as Codeの汎用エンジンです。Aigis のYAMLポリシーがエージェント特化の即時利用可能なソリューションであるのに対し、OPA/Cedarは既存のインフラ認可基盤との統合や、エージェント以外のワークロード（Kubernetesポリシー、API Gatewayなど）との一元管理に強みがあります。

大規模な組織では、OPA/Cedarをインフラ認可の基盤として採用し、Aigisをエージェント固有のガードレールとして併用する構成が現実的です。両者は競合するものではなく、レイヤーが異なります。

## まとめ

ガバナンス体制の構築は、一度に完璧を目指すものではありません。まず成熟度モデルで自社の現在地を把握し、ポリシー as Codeの技術基盤を導入し、AI CoE（またはバーチャルCoE）を設置する。このステップを踏むことで、Level 1〜2の「場当たり的」な状態からLevel 3〜4の「管理された」状態へ移行できます。

:::message
**2026年アップデート：ガバナンスの外部環境変化**

2026年初頭、ガバナンス体制の構築を加速させる複数の外部要因が発生しています。

- **NIST AI Agent Standards Initiative**（2026年2月）: Q4 2026にAI Agent Interoperability Profileの公開を予定。AIエージェントの安全性評価、相互運用性、テスト、ガバナンスに関する標準化が進行中。早期対応企業が認証取得・顧客信頼で先行できる
- **総務省 AIセキュリティ確保のための技術的対策ガイドライン**: ポリシーの「何をすべきか」に対して「どう実装するか」の技術ガイダンスを提供。本章のPolicy as Codeと併せて参照することで、実装レベルの抜け漏れを防げる
- **IPA 10大脅威2026**: AIリスクが組織向け脅威第3位にランクイン。経営層へのガバナンス投資の説得材料として最も有効
- **Cisco State of AI Security 2026**: 88%の組織がAIエージェント関連セキュリティインシデントを経験しているが、エージェントのアクセス範囲を可視化できている組織はわずか21%。ガバナンスの「認識と実態のギャップ」が最大のリスク要因

これらを踏まえ、本章で提示した5段階成熟度モデルにおいて、少なくとも**Level 3（Defined: ポリシー策定済み）**への到達を2026年8月（EU AI Act完全施行）までに目指すことを強く推奨します。
:::
