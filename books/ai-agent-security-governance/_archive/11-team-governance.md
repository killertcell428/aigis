# チームでのAIエージェントガバナンス

## 個人のベストプラクティス ≠ チームのガバナンス

前章ではAIコーディングエージェントの個別設定を解説しました。しかし、チームや組織でAIエージェントを運用する場合、**個人レベルの設定だけでは不十分**です。

チームガバナンスが必要な理由：

| 観点 | 個人利用 | チーム利用 |
|------|---------|-----------|
| **設定の一貫性** | 個人の裁量 | 全員が同じ基準で運用する必要 |
| **監査** | 自己責任 | 第三者による検証が必要 |
| **インシデント対応** | 個人で対処 | 組織的な対応フローが必要 |
| **知識共有** | 不要 | セキュリティ知識の標準化が必要 |
| **コンプライアンス** | 個人の判断 | 組織として法的要件を満たす必要 |

## ガバナンスフレームワーク

### Layer 1：ポリシー策定

組織としてのAIエージェント利用ポリシーを策定します。

```markdown
# AIエージェント利用ポリシー（テンプレート）

## 1. 適用範囲
本ポリシーは、業務において使用するすべてのAIエージェント
（Claude Code、GitHub Copilot、Cursor等）に適用される。

## 2. 利用承認
- AIエージェントの利用開始には、情報セキュリティ部門の承認が必要
- 新しいMCPサーバーの導入には、セキュリティレビューが必要

## 3. 機密情報の取り扱い
- 顧客データ、個人情報をAIエージェントの入力に含めてはならない
- APIキー、パスワード等の認証情報をAIとの対話で共有してはならない
- 社外秘の情報をAIに入力する場合は、承認済みの社内AIサービスのみ使用

## 4. 操作ログ
- すべてのAIエージェント操作はログに記録されなければならない
- ログは最低1年間保持される
- 月次でログの異常パターン分析を実施する

## 5. インシデント報告
- AIエージェントに関するセキュリティインシデントは、
  発見後24時間以内にセキュリティチームに報告する
```

### Layer 2：技術的統制

ポリシーを技術的に強制する仕組みを構築します。

```
技術的統制の構成要素:

1. 設定ファイルのリポジトリ管理
   → .claude/settings.json, CLAUDE.md をGitで管理
   → プルリクエストレビューで変更を承認

2. CI/CDパイプラインでの検証
   → セキュリティ設定ファイルの存在チェック
   → 認証情報の漏洩チェック（detect-secrets）
   → 依存関係の脆弱性チェック

3. ログ集約基盤
   → 全メンバーのAIエージェントログを集約
   → ダッシュボードで可視化
   → 異常パターンのアラート

4. APIキー管理
   → 個人別のAPIキー発行
   → 定期ローテーション
   → 使用量モニタリング
```

### Layer 3：運用プロセス

日常的な運用プロセスを定義します。

#### オンボーディング

```
新メンバー参画時:
1. AIエージェント利用ポリシーの説明・同意
2. 個人APIキーの発行
3. セキュリティ設定のインストール
   - .claude/settings.json
   - .claudeignore
   - pre-commit hooks
4. セキュリティ研修（30分）
   - 本書の主要リスクの概要
   - 禁止操作の説明
   - インシデント報告フロー
5. 設定の動作確認
```

#### 月次レビュー

```
月次セキュリティレビュー（1時間）:

1. ログ分析レポートの確認（15分）
   - 操作量の推移
   - 異常パターンの有無
   - ブロックされた操作の件数

2. インシデント振り返り（15分）
   - 発生したインシデント（あれば）
   - ヒヤリハット事例

3. 新しい脅威の共有（15分）
   - 今月公開されたCVE
   - 新しい攻撃手法
   - 本書の更新内容

4. 改善アクション（15分）
   - ポリシーの更新
   - 設定の調整
   - 研修の追加
```

#### インシデント対応

```
インシデント対応フロー:

Level 1（自動対応）:
  - 危険なコマンドのブロック（フックで自動）
  - 認証情報の検出とコミット防止（pre-commit）
  → アクション: ログ記録、ユーザーへの通知

Level 2（チーム対応）:
  - 不審な操作パターンの検出
  - 予期しない外部通信
  → アクション: セキュリティリーダーに報告、24時間以内に調査

Level 3（組織対応）:
  - データ漏洩の可能性
  - 認証情報の漏洩
  - 外部からの攻撃の兆候
  → アクション: CISO報告、全エージェント停止、フォレンジック
```

## Scope Creep（スコープクリープ）への対処

AIエージェントの特有のリスクとして、「頼んでいないことまでやってしまう」Scope Creepがあります。

```
例:
  依頼: 「テストを修正して」
  実際の動作: テスト修正 + 関連コードのリファクタリング + READMEの更新

  → 意図しない変更が本番コードに影響する可能性
```

### 対策

1. **CLAUDE.mdに明示的な制約を記載**: 「依頼された範囲のみを実行すること」
2. **差分レビューの徹底**: エージェントが生成した全変更をレビュー
3. **段階的な実行**: 大きなタスクは小さなステップに分割して実行

## 監査ダッシュボードの構築

```python
# 監査ダッシュボードのデータモデル例
class AuditDashboard:
    def daily_summary(self):
        return {
            "total_sessions": self.count_sessions(period="today"),
            "total_tool_calls": self.count_tool_calls(period="today"),
            "blocked_operations": self.count_blocked(period="today"),
            "external_communications": self.count_external_comms(period="today"),
            "unique_developers": self.count_unique_users(period="today"),
            "risk_alerts": self.get_alerts(period="today"),
        }
    
    def anomaly_detection(self):
        """過去30日の平均と比較して異常を検出"""
        today = self.daily_summary()
        avg_30d = self.average_summary(period="30d")
        
        anomalies = []
        if today["total_tool_calls"] > avg_30d["total_tool_calls"] * 3:
            anomalies.append("Tool call volume 3x above average")
        if today["external_communications"] > avg_30d["external_communications"] * 2:
            anomalies.append("External communications 2x above average")
        
        return anomalies
```

## ガバナンス成熟度モデル

組織のAIエージェントガバナンスの成熟度を4段階で評価します。

| レベル | 名称 | 特徴 |
|--------|------|------|
| **Level 0** | 未管理 | 各自が自由にAIエージェントを使用。ポリシーなし |
| **Level 1** | 基本管理 | ポリシーあり。CLAUDE.md/設定ファイルが標準化。ログ記録あり |
| **Level 2** | 積極的管理 | 自動化されたセキュリティチェック。月次レビュー。インシデント対応フロー |
| **Level 3** | 最適化 | リスクベースの権限管理。予測的な異常検知。継続的な改善プロセス |

多くの日本企業は現在Level 0またはLevel 1にあります。本章の内容を実装することで、Level 2への到達を目指しましょう。

## チェックリスト

```
【組織レベル】
□ AIエージェント利用ポリシーが策定・周知されている
□ APIキーが個人別に発行されている
□ ログ集約基盤が構築されている
□ インシデント対応フローが定められている
□ 月次セキュリティレビューが実施されている

【プロジェクトレベル】
□ セキュリティ設定ファイルがリポジトリに含まれている
□ pre-commitフックがインストールされている
□ MCPサーバーのホワイトリストが運用されている
□ コードレビューでAI生成コードが明示されている

【個人レベル】
□ セキュリティポリシーを理解・同意している
□ 個人APIキーを使用している
□ セキュリティ設定が正しく適用されている
□ インシデント報告フローを理解している
```
