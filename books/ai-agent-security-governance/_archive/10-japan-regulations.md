# 日本の法規制対応——総務省ガイドラインとコンプライアンス

## 日本のAIセキュリティ規制の全体像

日本におけるAI関連の規制・ガイドラインは、2025〜2026年にかけて急速に整備が進んでいます。

| 文書 | 発行元 | 性質 | 対象 |
|------|--------|------|------|
| **AIセキュリティ技術ガイドライン（案）** | 総務省 | 技術要件 | AI開発・運用者 |
| **AI事業者ガイドライン** | 内閣府 | 事業ガバナンス | AI事業者全般 |
| **個人情報保護法（改正）** | 個人情報保護委員会 | 法律 | 個人データ取扱者 |
| **金融分野AIガイドライン** | 金融庁 | 業界規制 | 金融機関 |
| **EU AI Act**（参考） | EU | 法律 | EU市場に提供するAI |

本章では特に、日本初の**技術的**AIセキュリティガイドラインである総務省ガイドラインを中心に解説します。

## 総務省「AIのセキュリティ確保のための技術的対策に係るガイドライン（案）」

### ガイドラインの位置づけ

- **正式名称**: AIのセキュリティ確保のための技術的対策に係るガイドライン（案）
- **公開**: 2025年12月（パブリックコメント受付）
- **特徴**: 日本初の**技術実装レベル**のAIセキュリティ基準（ガバナンスではなく技術）

従来の「AI事業者ガイドライン」が経営・ガバナンスレベルの指針であるのに対し、このガイドラインは**エンジニアが実装すべき具体的な技術要件**を定めています。

### 6つの脅威カテゴリ

ガイドラインが定義する6つの脅威カテゴリは、本書で解説してきた攻撃ベクトルと直結しています。

| # | 脅威カテゴリ | 本書の対応章 |
|---|-------------|-------------|
| 1 | **プロンプトインジェクション** | 第4章 |
| 2 | **データ汚染（RAG/学習データ）** | 第8章 |
| 3 | **モデル抽出（システムプロンプト窃取）** | 第3章（ASI07） |
| 4 | **サービス拒否（トークン枯渇）** | - |
| 5 | **エージェント誤動作（意図しない自律行動）** | 第12章 |
| 6 | **サプライチェーン攻撃（LLM API/MCP侵害）** | 第5,6,9章 |

### 8つの実装要件

#### 優先度1：高緊急度（今すぐ対応）

**要件1：入力検証・サニタイゼーション**

```python
# ガイドラインが求める入力検証の実装例
import re

INJECTION_PATTERNS = [
    r'ignore\s+(previous|prior|above)\s+instructions',
    r'you\s+are\s+now\s+a',
    r'system\s*:\s*',
    r'<\s*/?script',
    r'rm\s+-rf',
    r'curl\s+.*\|\s*sh',
    r'cat\s+/etc/(passwd|shadow)',
]

def validate_input(text: str) -> ValidationResult:
    """入力テキストの検証（ガイドライン要件1準拠）"""
    findings = []
    for pattern in INJECTION_PATTERNS:
        if re.search(pattern, text, re.IGNORECASE):
            findings.append(ValidationFinding(
                pattern=pattern,
                severity="high",
                recommendation="入力のサニタイゼーションまたは拒否"
            ))
    return ValidationResult(passed=len(findings) == 0, findings=findings)
```

**要件3：操作ログの記録**

ガイドラインでは以下の情報をログに記録することを求めています。

```python
# ガイドライン準拠のログフォーマット
log_entry = {
    "who": "user_id / session_id",        # 誰が
    "when": "2026-04-01T10:30:00+09:00",  # いつ
    "what": "file_read",                   # 何を
    "target": "/path/to/file",             # 対象
    "output_summary": "...",               # 結果の要約
    "risk_assessment": "low",              # リスク評価
}
```

ログ要件の詳細：
- **追記専用**（改変不可）
- **最低1年間**の保持を推奨
- **構造化フォーマット**（インシデント検索用）

**要件4：アクセス制御**

```yaml
# ガイドライン準拠のアクセス制御ポリシー（YAML形式）
access_control:
  principle: "least_privilege"  # 最小権限
  
  roles:
    developer:
      file_access: ["./src/", "./tests/"]
      commands: ["npm test", "git *"]
      network: ["localhost", "registry.npmjs.org"]
    
    reviewer:
      file_access_readonly: ["./src/", "./tests/", "./docs/"]
      commands: ["git log", "git diff"]
      network: []
    
    deployer:
      requires_approval: true
      file_access_readonly: ["./dist/"]
      commands: ["npm run deploy"]
      network: ["deploy.company.com"]
```

**要件7：インシデントレスポンス計画**

```
インシデント対応フロー:
1. 検知  → ログ監視、異常パターンアラート
2. 停止  → エージェントの即時停止
3. 隔離  → 影響範囲の特定と隔離
4. 調査  → ログベースの原因分析
5. 復旧  → 汚染データの除去、設定の修正
6. 報告  → ステークホルダーへの報告
7. 改善  → 再発防止策の実装
```

#### 優先度2：中緊急度（3ヶ月以内）

**要件2：出力監視**

```python
# PII（個人情報）の検出
PII_PATTERNS = [
    r'\d{3}-\d{4}-\d{4}',       # 電話番号
    r'\d{3}-\d{4}',              # 郵便番号
    r'[A-Z]{2}\d{6}',           # パスポート番号パターン
    r'\d{4}-\d{4}-\d{4}-\d{4}', # クレジットカード番号
]

# 危険なコード生成の検出
DANGEROUS_CODE_PATTERNS = [
    r'rm\s+-rf\s+[/*]',         # ファイル全削除
    r'curl\s+.*\|\s*(sh|bash)',  # リモートスクリプト実行
    r'eval\s*\(',               # 動的コード実行
    r'exec\s*\(',               # プロセス実行
]
```

## EU AI Actとの対応関係

日本企業がEU市場にAIサービスを提供する場合、EU AI Actへの対応も必要です。

| EU AI Act 要件 | 総務省ガイドライン | 対応状況 |
|----------------|-------------------|----------|
| リスク評価 | 脅威カテゴリの定義 | ○ カバー |
| ログ記録 | 要件3（操作ログ） | ○ カバー |
| 透明性 | 要件2（出力監視） | △ 部分的 |
| 人間の監視 | 要件4（アクセス制御） | ○ カバー |
| セキュリティ | 要件1,5,6 | ○ カバー |
| 品質管理 | 要件8（定期評価） | ○ カバー |

## 対応ロードマップ

```
Phase 1: 基盤整備（1-2ヶ月）
├── 要件1: 入力検証の実装
├── 要件3: ログ記録の開始
├── 要件4: アクセス制御ポリシーの策定
└── 要件7: インシデントレスポンス計画の策定

Phase 2: 防御強化（3-4ヶ月）
├── 要件2: 出力監視の実装
├── 要件6: サプライチェーン管理の開始
└── ログ基盤の統合（SIEM連携）

Phase 3: 成熟化（5-6ヶ月）
├── 要件5: RAGセキュリティの実装
├── 要件8: 定期評価プロセスの確立
└── コンプライアンスレポートの自動生成
```
