# A2A・メモリ汚染・サプライチェーン——拡大する攻撃面

前章まではプロンプトインジェクションとMCPに焦点を当てました。本章では、AIエージェントの攻撃面をさらに広げる3つの脅威を解説します。

---

## 1. A2Aプロトコルのセキュリティ

### A2Aとは

A2A（Agent-to-Agent）プロトコルは、Googleが提唱したエージェント間の通信標準です。MCPが「エージェント↔ツール」の接続を担うのに対し、A2Aは「エージェント↔エージェント」の連携を可能にします。

```
MCPの世界:  エージェント → MCPサーバー → ツール/データソース
A2Aの世界:  エージェントA ←→ エージェントB ←→ エージェントC
```

マルチエージェントシステムが実現する一方、攻撃面は指数的に拡大します。

### 4つの主要攻撃ベクトル

**❶ Spoofed Agent Card（偽装エージェントカード）**

A2Aでは各エージェントが「Agent Card」で能力を宣言します。攻撃者がDNSポイズニング等で偽のAgent Cardを配布し、正当なエージェントになりすまします。

```
正常: エージェントA → DNS → 正規B → タスク委任
攻撃: エージェントA → 汚染DNS → 偽B → 攻撃者にタスク委任
```

**❷ Task Injection（タスク注入）**

タスク委任ペイロードに隠されたコマンドを埋め込む攻撃です。エージェント間のプロンプトインジェクションとも言えます。

**❸ Broken Trust Chain（信頼チェーンの断絶）**

A→B→Cの多段委任でエンドツーエンドの署名がないため、中間エージェントBがメッセージを改竄できます。

**❹ Agent Card Poisoning（レジストリ汚染）**

エージェントカタログに虚偽のAgent Cardを登録し、タスクを横取りする攻撃です。

### A2A + MCP の複合リスク

A2AとMCPが組み合わさると、プロトコル横断型攻撃が可能になります。A2Aレイヤーの認証を通過した通信が、MCP側では正当なリクエストに見えるため、監視の死角が生まれます。

### 対策の方向性

| 脅威 | 対策 | 優先度 |
|------|------|--------|
| 偽装Agent Card | ETDIベースの検証 + フィンガープリント管理 | 高 |
| タスク注入 | PI検知 + コンテキスト境界 | 高 |
| 信頼チェーン断絶 | エンドツーエンド署名 + タスクID追跡 | 中 |
| レジストリ汚染 | ホワイトリスト型レジストリ | 中 |
| MCP+A2A複合攻撃 | プロトコル横断のイベント相関ログ | 高 |

:::message
A2Aプロトコルはまだ初期段階です。**信頼できるエージェントのみとの通信に制限**し、仕様の更新に追従することが現時点での最善策です。
:::

## 2. AIメモリ汚染攻撃

### 一時的な攻撃 vs 永続的な攻撃

プロンプトインジェクションはセッション内でのみ有効です。しかし**メモリ汚染（Memory Poisoning）**は、RAGのベクトルDBやエージェントの長期メモリに悪意あるデータを注入し、**永続的に**判断を歪めます。

![メモリ汚染の全体像](/images/ch08-memory-poisoning.png)
*図：AIメモリ汚染の攻撃フローと被害統計*

### 攻撃フロー

```
❶ 挿入: 攻撃者がWebページ/ドキュメントに非表示テキストを埋め込む
         （white-on-white、CSS hidden、HTMLメタタグ）
         ↓
❷ 吸収: RAGの自動クロールで非表示テキストがベクトルDBに取り込まれる
         ↓
❸ 永続化: 汚染データがDBに格納。以後のすべてのクエリに影響
         ↓
❹ 拡散: AIが繰り返し利用することで汚染が「知識」として強化
```

### 被害の実態

- **Microsoft Security Blog（2026年2月）**: **50社以上**が既に被害確認
- **Galileo AI研究**: 単一の汚染で**4時間以内に意思決定の87%**が影響
- 感染はAIの利用ごとに拡散・強化される

### なぜ検知・除去が困難か

| 要因 | 詳細 |
|------|------|
| 信頼ソースに偽装 | 汚染データは正規ドキュメントの一部として取り込まれる |
| 推論の不透明さ | なぜ推奨が変わったかをLLM自身が説明できない |
| 緩やかなドリフト | 急激な変化ではなく徐々に偏る |
| 除去の副作用 | 汚染ベクトルの除去で正当なデータも損傷 |

### 実世界の攻撃シナリオ

- **競合製品のレピュテーション操作**: 社内AIが特定製品を不当に推奨
- **セキュリティ判断の歪曲**: 攻撃者のインフラを「安全」と判定
- **企業意思決定の操作**: 市場レポートの偏りによる投資判断の歪み

### 防御の優先度

| 対策 | 難易度 | 効果 |
|------|--------|------|
| **ソース制限（ホワイトリスト）** | 低 | 高 |
| **非表示コンテンツ検出** | 中 | 中 |
| **RAGストア定期スキャン** | 中 | 中 |
| **トレーサビリティ（ソースログ）** | 高 | 高 |
| **ヒューマンレビューゲート** | 中 | 高 |

## 3. サプライチェーン攻撃

### AIエージェントのサプライチェーン

AIエージェントは多くの外部コンポーネントに依存しています。

```
依存関係の連鎖:
  LLMプロバイダーAPI → AIフレームワーク → MCPサーバー
  → Pythonライブラリ → Dockerイメージ → CI/CDパイプライン
```

いずれか1つが侵害されれば、エージェント全体のセキュリティが崩壊します。

### 実例：litellm マルウェア事件

2025年、LLMプロキシライブラリ「LiteLLM」のv1.82.7/1.82.8に、マルウェアが混入した状態でPyPIに公開されました。メンテナーアカウントが侵害され、正規ソースコードに改変を加えたパッケージが配布されました。

### 実例：Postmark MCPバックドア

完全に機能するバックドア付きMCPサーバーが公開されていた事例。表面上は正当なメール送信ツールとして機能しながら、裏でデータを外部送信していました。

### 実例：CVE-2025-6514（mcp-remote）

MCPリモート接続用パッケージにOSコマンドインジェクション。CVSS 9.6。**43万7000以上の環境**に影響。

### 防御策

```
即時対応:
  ✅ ロックファイル（package-lock.json / uv.lock）のコミット
  ✅ バージョンの厳密固定（^ や ~ を使わない）
  ✅ MCPサーバー導入前のソースコード確認

自動化:
  ✅ CI/CDでの定期脆弱性スキャン（npm audit / pip-audit）
  ✅ Dependabot/Renovateの導入
  ✅ パッケージ署名の検証（npm audit signatures）

方針:
  ✅ 最小依存の原則（不必要なパッケージを追加しない）
  ✅ 開発依存と本番依存の分離
  ✅ GitHub ActionsのアクションをSHAで固定
```

## 3つの脅威に共通する防御の原則

これらの脅威に共通するのは、**AIエージェントの「接続性」が攻撃面を拡大する**という構造です。

| 原則 | 適用 |
|------|------|
| **最小接続** | 必要なエージェント・ソース・依存関係のみに限定 |
| **検証ベース** | 信頼を前提とせず、すべてを検証 |
| **ログ記録** | プロトコル横断のイベント相関で異常を検知 |
| **定期監査** | RAGストア・依存関係・ツール定義の定期チェック |

次のSection 2では、これらの脅威に対する組織的な対策設計を、実際のインシデント事例を起点に解説します。
