# 企業導入ロードマップ——フェーズ別の実装計画

## 導入の現実

AIエージェントのセキュリティを一度にすべて整備することは、現実的ではありません。本章では、企業がAIエージェントを安全に導入・拡大するための**段階的なロードマップ**を提示します。

![企業導入ロードマップ](/images/ch17-roadmap.png)
*図：企業導入ロードマップ — フェーズ別の実装計画*

## フェーズ0：現状把握（1〜2週間）

すべてはここから始まります。

### やるべきこと

```
1. インベントリ作成
   □ 組織内で使用されているAIエージェントの一覧
     - Claude Code、GitHub Copilot、Cursor、その他
   □ 使用しているMCPサーバーの一覧
   □ 使用しているAPIキーの一覧と所有者
   □ AIエージェントがアクセスしているデータ・システムの一覧

2. リスク評価
   □ 各AIエージェントの権限範囲の確認
   □ 機密データへのアクセス可能性の確認
   □ 現在のログ記録状況の確認
   □ OWASP Agentic Top 10に基づく脅威の洗い出し

3. ギャップ分析
   □ 総務省ガイドラインの8要件との対応状況
   □ 既存のIAMポリシーとAIエージェントの整合性
   □ インシデントレスポンス計画のAI対応状況
```

### 成果物

- AIエージェント利用状況レポート
- リスクマトリクス（脅威×影響度×現状対策）
- 優先対応事項リスト

## フェーズ1：基盤構築（1ヶ月）

最も緊急度の高い対策を実装します。

### 1-1. ポリシーの策定

```markdown
策定すべきポリシー:
1. AIエージェント利用ポリシー
2. MCPサーバー承認プロセス
3. APIキー管理ポリシー
4. インシデントレスポンス計画（AI対応版）
```

### 1-2. 基本的なセキュリティ設定

```
全プロジェクトに適用:
├── CLAUDE.md（セキュリティポリシー記載）
├── .claudeignore（機密ファイル除外）
├── .claude/settings.json（権限設定）
├── .pre-commit-config.yaml（認証情報検出）
└── .gitignore（追加の除外パターン）
```

### 1-3. ログ記録の開始

```
最低限のログ記録:
- 誰が（ユーザーID）
- いつ（タイムスタンプ）
- 何を（ツール名、操作）
- 対象（ファイルパス、URL等）
- 結果（成功/失敗/ブロック）
```

### 1-4. APIキーの分離

```
即時実行:
- チーム共有キー → 個人別キーに分離
- 用途別キー分離（開発用/CI用/本番用）
- 使用量モニタリングの開始
```

### マイルストーン

- [ ] 全プロジェクトにセキュリティ設定ファイルが存在する
- [ ] 全メンバーが個人APIキーを使用している
- [ ] 操作ログが記録されている
- [ ] インシデント対応フローが文書化されている

## フェーズ2：防御強化（2〜3ヶ月）

自動化された防御機能を導入します。

### 2-1. 入出力フィルタリング

```python
# Aigisまたは同等ツールの導入
from aigis import Guard

guard = Guard()
# アプリケーションのLLM呼び出しにフィルタリングを統合
```

### 2-2. フックの自動化

```json
// Claude Codeフックの組織標準化
{
  "hooks": {
    "PreToolUse": [
      {"matcher": "Bash", "command": "security_hooks/check_command.py"},
      {"matcher": "Write", "command": "security_hooks/check_file.py"}
    ],
    "PostToolUse": [
      {"matcher": "*", "command": "security_hooks/audit_log.py"}
    ]
  }
}
```

### 2-3. CI/CDセキュリティ統合

```yaml
# GitHub Actions でのセキュリティチェック
- name: Security Scan
  steps:
    - npm audit --audit-level=high
    - pip-audit
    - detect-secrets scan
    - aigis scan ./
```

### 2-4. MCPサーバー管理

```
MCPサーバー管理プロセス:
1. 導入申請 → セキュリティレビュー → 承認/却下
2. 承認済みサーバーのホワイトリスト管理
3. 月次の脆弱性チェック
4. tool description の定期検証
```

### マイルストーン

- [ ] 入出力フィルタリングが全アプリケーションに導入されている
- [ ] セキュリティフックが全プロジェクトに展開されている
- [ ] CI/CDに脆弱性スキャンが統合されている
- [ ] MCPサーバーの承認プロセスが運用されている

## フェーズ3：ガバナンス成熟化（4〜6ヶ月）

組織的なガバナンス体制を確立します。

### 3-1. NHI管理の導入

```
NHI管理の展開:
1. AIエージェントセッションのIAM統合
2. セッション単位の一時認証（JIT）
3. 権限の定期レビュー
4. アイデンティティのライフサイクル管理
```

### 3-2. SIEM統合

```
ログ基盤の高度化:
1. 全AIエージェントログをSIEMに集約
2. 異常検知ルールの設定
3. ダッシュボードの構築
4. アラートの設定
```

### 3-3. コンプライアンス対応

```
規制対応:
1. 総務省ガイドラインの全8要件への対応完了
2. コンプライアンスレポートの自動生成
3. 定期的な監査プロセスの確立
4. EU AI Act対応（該当する場合）
```

### 3-4. セキュリティ文化の醸成

```
継続的な取り組み:
1. 月次セキュリティレビューの定例化
2. AIセキュリティ研修の四半期実施
3. インシデント振り返りの共有
4. 外部セキュリティ評価の年次実施
```

### マイルストーン

- [ ] NHI管理がIAMに統合されている
- [ ] SIEMで全AIエージェントログが監視されている
- [ ] 総務省ガイドラインの全8要件に対応完了
- [ ] 四半期のセキュリティ研修が定例化されている

## 投資対効果（ROI）の考え方

セキュリティ投資の必要性を経営層に説明する際のフレームワーク：

```
コスト（セキュリティ未整備の場合）:
├── データ漏洩のコスト
│   → IBM「Cost of a Data Breach 2025」: 平均$4.88M
│   → AIが関与する漏洩は+20%のコスト増
├── コンプライアンス違反の罰金
│   → EU AI Act: 最大€35M or 全世界売上7%
├── レピュテーション損害
│   → 顧客信頼の喪失、株価影響
└── 事後対応コスト
    → フォレンジック、法的対応、通知費用

コスト（セキュリティ整備の場合）:
├── ツール費用: 年$1,000〜$10,000程度
├── 設定・導入工数: 1〜2人月
├── 運用工数: 月4〜8時間（レビュー・更新）
└── 研修工数: 四半期で半日
```

## よくある失敗パターンと対策

| 失敗パターン | 対策 |
|-------------|------|
| 「AIは社内利用だから安全」 | 内部脅威・意図しない操作のリスクを説明 |
| 「セキュリティは生産性を下げる」 | 適切な設定は作業の自由度を下げない |
| 「全部一度に導入しようとして頓挫」 | フェーズ別の段階的導入 |
| 「設定して放置」 | 月次レビューの定例化 |
| 「IT部門だけの問題」 | 全メンバーの意識向上が必要 |

## 経営層への提言

```
AIエージェントは業務効率を飛躍的に向上させますが、
セキュリティなしでの導入は、以下のリスクを抱えます。

1. 機密データの漏洩（顧客情報、営業秘密）
2. 法規制違反（総務省ガイドライン、個人情報保護法）
3. サイバー攻撃の侵入経路化（APT攻撃の実例あり）
4. 意図しないシステム操作による業務停止

対策のコストは、インシデント発生時のコストに比べて
桁違いに小さいものです。
段階的な導入により、セキュリティと生産性を両立できます。
```
