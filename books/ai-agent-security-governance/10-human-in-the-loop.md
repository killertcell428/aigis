# Human-in-the-Loopの設計

## 3つの基本パターン

AIエージェントの自律性が高まるほど、「どの時点で人間が介入するか」の設計が重要になります。Human-in-the-Loop（HITL）は単なる承認フローではなく、エージェントの自律性と人間の統制のバランスを定義するアーキテクチャ設計です。

実務において、HITLは以下の3つの基本パターンに分類できます。

### パターン1: Approval Gate（承認ゲート）

エージェントが特定のアクションを実行する前に、必ず人間の承認を得るパターンです。最もシンプルで最も安全性が高い方式ですが、エージェントの応答速度が人間の承認待ち時間に制約されます。

```
エージェント → アクション計画を提示 → 人間が承認/却下 → 実行 or 中止
```

**適用シーン**: 本番データベースへの書き込み、外部APIへのデータ送信、金額が一定額を超える取引の実行など、不可逆性が高いアクション。

Approval Gateは「常にすべてのアクションを承認する」運用にすると、後述するラバースタンプ問題を引き起こします。リスクの高いアクションのみに限定し、低リスクアクションは自律実行を許可する設計が重要です。

### パターン2: Escalation Trigger（エスカレーショントリガー）

エージェントが自律的に行動しつつ、事前定義された条件に合致した場合にのみ人間にエスカレーションするパターンです。日常的なタスクの効率を維持しながら、異常やリスクを人間がハンドリングします。

```
エージェント → 自律実行 → 条件合致 → 人間にエスカレーション → 判断 → 継続 or 中止
```

**トリガー条件の例**:
- 信頼度スコアが閾値を下回った場合
- ガードレール違反が検出された場合
- 未知のツールやAPIへのアクセス要求
- エラーの連続発生（3回以上のリトライ）
- ポリシーで`action: review`と定義されたアクション

### パターン3: Tiered Escalation（段階的エスカレーション）

リスクレベルに応じてエスカレーション先を変える多層構造です。エンタープライズ環境では最も現実的なパターンです。

```
リスクレベル    対応
──────────    ──────────────────────
Low           自律実行（ログ記録のみ）
Medium        チーム内承認（Slack通知→担当者承認）
High          マネージャー承認 + セキュリティレビュー
Critical      CISO / AI CoE 承認 + 監査証跡の即時生成
```

段階的エスカレーションを設計する際には、リスクレベルの判定基準を明確に定義し、ポリシー as Codeとして実装する必要があります。曖昧な判定基準はエスカレーションの遅延や見逃しにつながります。

## 制度的要件（EU AI Act / NIST / 日本）

HITLは技術的なベストプラクティスであるだけでなく、主要な規制フレームワークにおいて法的要件としても位置付けられています。

### EU AI Act 第14条

EU AI Act第14条は、高リスクAIシステムに対して人間の監視（human oversight）を義務付けています。具体的には、監視者がAIシステムの出力を**無視**し、**上書き**し、あるいは**停止**する権限を持つことが求められます。

これはApproval Gateの法的根拠に直結します。高リスクに分類されるAIエージェントでは、人間が実行前にアクションを差し止める能力を技術的に実装しなければなりません。形式的な承認ボタンではなく、エージェントの行動を実質的に制御する権限が必要です。

EU AI Actは2026年8月に完全施行されますが、高リスクAI要件は段階的に適用が開始されます。EU市場でAIエージェントを提供する企業は、HITLの実装を含むコンプライアンス対応を計画的に進める必要があります。

### NIST AI Risk Management Framework

NIST AI RMF（AI Risk Management Framework）はHITLを複数の観点から要求しています。**GOVERN**機能ではAIシステムの監視体制の確立を、**MAP 1.6**では人間監視の必要性の特定を、**MEASURE 2.6**では人間監視の有効性の測定を、それぞれ求めています。

NISTのフレームワークは法的拘束力を持ちませんが、米国政府調達における事実上の標準であり、多くの民間企業もリスク管理のベースラインとして採用しています。

### 日本のAI事業者ガイドライン

日本のAI事業者ガイドラインv1.2（2026年3月公開）は、AIエージェントに初めて対応した改訂版です。同ガイドラインでは、HITLは事実上必須とされており、自律的に行動するAIシステムにおける人間の監視・介入メカニズムの実装を求めています。

法的拘束力はソフトローの位置付けですが、「ガイドラインに準拠していない」ことがインシデント発生時の過失認定に影響する可能性があるため、実質的な遵守が求められます。

## ラバースタンプ問題への対処

HITLの最大の課題は、人間の承認が形骸化する「ラバースタンプ問題」です。これはOWASP Agentic Top 10のASI07「Uncontrolled Autonomous Actions」やASI09「Insufficient Logging and Monitoring」に関連するリスクでもあります。人間が監視しているはずなのに実質的に機能していない——これが最も危険な状態です。

### 自動化バイアスの実態

Parasuraman & Manzey（2010, Human Factors）の研究は、人間が自動化システムの出力を批判的に評価せず、無批判に受け入れる傾向——自動化バイアス（automation bias）——を体系的に示しました。この傾向はAIエージェントの承認プロセスでも顕著に現れます。

Goddard et al.（2012, JAMIA）は医療分野の臨床意思決定支援システム（CDSS）において、自動化バイアスが診断エラーにつながる事例を報告しています。AIの判断を人間が「確認」するフローを設けても、確認が実質的に機能しないのです。

エージェントの承認フローでも同様の問題が発生します。1日に数十回、数百回と承認リクエストが届けば、人間は内容を精査せずに「承認」をクリックするようになります。これはHITLが存在しないのと実質的に同じです。

### 対策アプローチ

ラバースタンプ問題に対処するためには、以下の多層的な対策が必要です。

**情報負荷の最適化**: 承認リクエストには、エージェントのアクション内容、影響範囲、リスクレベルを簡潔に提示します。大量の生データを人間に読ませるのではなく、判断に必要な情報を構造化して提示することが重要です。

**ランダム詳細レビュー**: 承認フローの一定割合（例: 10%）で、詳細なレビューを強制する仕組みを導入します。承認者は追加の確認項目に回答しなければ承認を完了できません。これにより、ルーチン的な承認に緊張感が生まれます。

**承認メトリクスの監視**: 承認率、承認までの時間、却下理由の分布を定量的に監視します。承認率が99%を超え、承認時間が数秒以内であれば、ラバースタンプ化している可能性が高いと判断できます。

**ローテーションと教育**: 承認者を定期的にローテーションし、新鮮な視点を確保します。また、実際のインシデント事例や攻撃パターンを用いた教育を定期的に実施し、「なぜ承認が重要か」への理解を維持します。

## 実装アーキテクチャ

### ワークフローエンジン統合

HITLの実装において、ワークフローエンジンとの統合は最も堅牢なアプローチです。

**Temporal**はワークフローをコードとして定義でき、人間の承認をワークフロー内の「待機ステップ」として自然に表現できます。タイムアウト、リトライ、補償トランザクションが組み込みでサポートされるため、承認が一定時間内に得られない場合の代替フローも容易に実装できます。

**Apache Airflow**はデータパイプラインの管理で広く使われており、DAG（有向非巡回グラフ）内に人間の承認ノードを配置できます。既存のデータ基盤と統合する場合に適しています。

### Slack/Teams承認フロー

多くの組織では、Slack/Microsoft Teamsのインタラクティブメッセージ機能を使った承認フローが最も導入障壁が低い方式です。

```
[エージェント] → Slack通知:
  ┌─────────────────────────────────┐
  │ 🔔 承認リクエスト                   │
  │                                   │
  │ アクション: 外部API呼び出し          │
  │ 対象: analytics-api.example.com    │
  │ リスクレベル: Medium               │
  │ データ: 匿名化済み集計データ         │
  │                                   │
  │ [✅ 承認]  [❌ 却下]  [📋 詳細]     │
  └─────────────────────────────────┘
```

この方式は手軽ですが、Slackの通知が他のメッセージに埋もれるリスクがあるため、承認待ちのSLAとエスカレーションルール（例: 30分以内に承認されなければマネージャーにエスカレーション）を定義しておく必要があります。

### LangGraphによる実装

LangGraphはLangChain上に構築されたエージェントオーケストレーションフレームワークで、HITLをグラフのノードとして自然に組み込めます。

```python
from langgraph.graph import StateGraph

def human_approval_node(state):
    """人間の承認を待つノード"""
    action = state["pending_action"]
    risk_level = assess_risk(action)

    if risk_level == "low":
        return {"approved": True, "reviewer": "auto"}

    # Medium/High: 人間の承認を要求
    approval = request_human_approval(
        action=action,
        risk_level=risk_level,
        timeout_seconds=1800,  # 30分タイムアウト
    )
    return {"approved": approval.decision, "reviewer": approval.user}

graph = StateGraph(AgentState)
graph.add_node("plan", plan_action)
graph.add_node("review", human_approval_node)
graph.add_node("execute", execute_action)
graph.add_edge("plan", "review")
graph.add_conditional_edges(
    "review",
    lambda s: "execute" if s["approved"] else "plan",
)
```

LangGraphのhuman-in-the-loopノードは、承認待ちの間に状態をチェックポイントとして永続化するため、プロセスの再起動やスケールアウトにも対応できます。

### 監査証跡の自動生成

すべてのHITLイベントは監査証跡として記録する必要があります。記録すべき項目は以下のとおりです。

| 項目 | 内容 |
|---|---|
| タイムスタンプ | 承認リクエストの発行時刻と承認/却下時刻 |
| アクション詳細 | エージェントが実行しようとしたアクションの内容 |
| リスクレベル | ポリシーエンジンが判定したリスクレベル |
| 承認者 | 承認/却下した人物の識別子 |
| 判断理由 | 却下の場合はその理由（自由記述 or 選択式） |
| 実行結果 | 承認後のアクション実行結果 |

これらの監査証跡は、次章で解説するオブザーバビリティ基盤に統合され、コンプライアンスレポートの素材となります。

### コスト意識を持った設計

HITLの実装コストは、承認頻度に比例します。すべてのアクションにApproval Gateを設ければ安全性は最大化しますが、承認者の工数が膨大になり、エージェント導入の費用対効果が損なわれます。

以下の設計指針で、安全性と効率のバランスを取ります。

**リスクベースの承認設計**: エージェントのアクションをリスクレベルで分類し、高リスクアクションのみ承認を要求します。低リスクアクション（読み取り専用のクエリ、社内データの参照等）は自律実行を許可し、ログ記録で監査性を確保します。

**承認の自動期限切れ**: 一定時間内に承認が得られない場合のデフォルト動作を定義します。安全側に倒す設計（タイムアウト=却下）が基本ですが、低リスクのアクションではタイムアウト=自動承認とすることで、承認者不在時のブロックを防げます。

**承認キャッシュ**: 同一の承認者が同一パターンのアクションを繰り返し承認する場合、一定期間（例: 1時間）は再承認を省略する仕組みです。ラバースタンプ問題を助長するリスクがあるため、高リスクアクションには適用しません。

## まとめ

HITLの設計は「承認ボタンを置けば完了」ではありません。3つの基本パターンから自社に適切なモデルを選択し、規制要件を満たし、ラバースタンプ問題に対処する——この一連の設計が、AIエージェントの「統制された自律性」を実現します。

HITLはAigisの直接的な対応範囲外ですが、Aigisのポリシーエンジンで`action: review`を定義することで、Escalation Triggerの発動条件をポリシー as Codeとして管理できます。ワークフローエンジン（Temporal/Airflow）やSlack/Teams統合と組み合わせることで、実運用可能なHITL基盤を構築してください。

:::message
**2026年アップデート：HITL必須化の動き**

2026年に入り、HITLの設計は「ベストプラクティス」から「規制要件」へと格上げされつつあります。

- **日本AI事業者ガイドラインv1.2**（2026年3月31日）: 高リスク領域でのエージェント利用に対し、HITL（Human-in-the-Loop）を**事実上の必須要件**として規定。「AIエージェントが外部システムに影響を与えるアクションを実行する場合、人間の判断介在メカニズムが必要」と明記
- **EU AI Act第14条**: 高リスクAIシステムへの人間による監視を法的義務として規定（2026年8月完全施行）
- **OWASP Agentic Top 10 2026 ASI09**: 「Human-Agent Trust Exploitation」として、ユーザーが自信ありげに聞こえるエージェントを過信する問題を独立リスクとして認定。適切なHITL設計はこのリスクへの直接的な対策

本章で解説した3パターン（Approval Gate、Escalation Trigger、Tiered Escalation）のうち、最低限**Escalation Trigger**パターンの導入は、規制対応として必須の水準に到達しています。
:::
