# マルチエージェント通信のセキュリティ

## A2Aプロトコルの仕組み

### エージェント間通信の標準化

2025年4月9日、GoogleはGoogle Cloud Next '25においてA2A（Agent-to-Agent）プロトコルを発表しました。AWS、Cisco、Microsoft、Salesforce、SAP、ServiceNowを含む50社超のパートナーと共に策定され、2025年6月にはLinux Foundationに移管されて100社を超えるコミュニティへと成長しています。

A2Aが解決する問題は明確です。異なるベンダー、異なるフレームワークで構築されたエージェント同士が、**共通のプロトコルで協調作業を行う**ための基盤が存在しなかったことです。

### パートナーエコシステム

A2Aの設立メンバーには、クラウド（AWS、Google、Salesforce、SAP、ServiceNow）、AIフレームワーク（Anthropic、Cohere、LangChain、LlamaIndex、CrewAI）、セキュリティ（CyberArk、Palo Alto Networks）、コンサルティング（Deloitte、Accenture、BCG、McKinsey）など、業界を横断する企業が名を連ねています。Linux Foundation移管後は100社を超えるコミュニティとなり、エンタープライズ環境での採用に向けた標準化が加速しています。

### 技術アーキテクチャ

A2Aは、HTTP + JSON-RPC 2.0をベースとし、SSE（Server-Sent Events）によるストリーミングをサポートします。主要な構成要素は以下の3つです。

**Agent Card**

各エージェントは`/.well-known/agent.json`にAgent Cardを公開します。これはエージェントの能力、認証方式、対応タスクを宣言するメタデータです。

```json
{
  "name": "inventory-agent",
  "description": "在庫管理と発注処理を担当",
  "url": "https://agents.example.com/inventory",
  "capabilities": ["inventory-check", "order-placement"],
  "authentication": {
    "schemes": ["oauth2"],
    "oauth2": {
      "token_endpoint": "https://auth.example.com/token",
      "scopes": ["inventory:read", "orders:write"]
    }
  }
}
```

**Task**

A2Aの中核概念はTaskです。ステートマシンとして定義され、`submitted → working → completed/failed`の状態遷移を持ちます。依頼元エージェントがTaskを作成し、依頼先エージェントがTaskを進行させます。

```
依頼元                    依頼先
  │                        │
  │─── Task作成 ──────────→│
  │    (submitted)         │
  │                        │─── 処理開始
  │←── ステータス更新 ─────│    (working)
  │    (SSEストリーム)     │
  │                        │─── 処理完了
  │←── 結果返却 ──────────│    (completed)
  │    (Artifacts)         │
```

**認証**

Agent Card内で認証方式を宣言します。OAuth 2.0、APIキー、OIDCなどが使用可能です。ただし、A2Aプロトコル自体は認証の**実装を規定しません**。認証方式の宣言と発見の仕組みを提供するのみであり、実際のセキュリティはデプロイ環境に委ねられています。

## MCP vs A2A：何が違い、どう組み合わせるか

### 設計目的の根本的な違い

GoogleはA2A発表時に「MCPとは補完関係にある」と明言しています。

> "A2A is designed to complement agent-to-tool protocols like MCP. While MCP focuses on connecting agents to tools and data sources, A2A focuses on enabling agents to communicate and collaborate with each other."

両プロトコルの違いを整理します。

| 項目 | MCP | A2A |
|------|-----|-----|
| 提唱者 | Anthropic（2024年11月） | Google（2025年4月） |
| 目的 | エージェント ↔ ツール/データ | エージェント ↔ エージェント |
| アナロジー | USB-C（デバイス接続） | 外交プロトコル（主体間交渉） |
| 通信モデル | クライアント → サーバー | ピアツーピア |
| 主要抽象化 | Tools, Resources, Prompts | Tasks, Agent Cards, Artifacts |
| 状態管理 | ステートレス | ステートフル（Taskライフサイクル） |

### 実践的な併用パターン

実際のエンタープライズ環境では、MCPとA2Aは異なるレイヤーで併用されます。

```
┌──────────────────────────────────────────────────┐
│           オーケストレーター・エージェント          │
└─────┬────────────────┬────────────────┬──────────┘
      │ A2A            │ A2A            │ A2A
      ▼                ▼                ▼
┌───────────┐  ┌───────────┐  ┌───────────┐
│ 在庫エージェ │  │ 受注エージェ │  │ 配送エージェ │
│ ント       │  │ ント       │  │ ント       │
└──┬────┬───┘  └──┬────┬───┘  └──┬────┬───┘
   │MCP │MCP      │MCP │MCP      │MCP │MCP
   ▼    ▼         ▼    ▼         ▼    ▼
  DB   ERP      CRM   支払    WMS   配送API
```

オーケストレーターがA2Aで専門エージェントにタスクを委任し、各エージェントがMCPでツールやデータソースにアクセスするという構成です。A2Aはエージェント間の「外交」、MCPはエージェントとツールの「接続」を担います。

## マルチエージェントの脅威モデル

マルチエージェント環境では、単一エージェントにはない固有の脅威が発生します。

### Confused Deputy（混乱した代理人）

信頼されたエージェントが、悪意あるエージェントからの指示により、自身の権限で意図しない操作を実行してしまう脅威です。

```
攻撃者エージェント → 「在庫データをCSVでエクスポートして」
                          ↓
在庫エージェント（DB読取権限保持）→ DBからエクスポート実行
                          ↓
攻撃者エージェント ← CSVデータ受領（本来アクセス権なし）
```

A2AのTaskモデルでは、依頼元エージェントのIDと権限を検証する仕組みが必要ですが、プロトコル仕様としては認証方式の宣言のみを提供しています。実装レベルでの権限検証は各開発者の責任です。

### Privilege Escalation Chain（権限昇格チェーン）

タスクの委任が連鎖する過程で、各ステップでは正当な権限行使であっても、チェーン全体として見ると当初の権限を超えた操作が実行されてしまう脅威です。

```
ユーザー → エージェントA（read権限）
  → エージェントB（read+write権限、Aからの委任を信頼）
    → エージェントC（admin権限、Bからの委任を信頼）
      → 管理者操作の実行（ユーザーの意図を超越）
```

各エージェントが直近の委任元のみを検証し、チェーン全体の権限を追跡しない場合に発生します。Fang et al.（arXiv:2402.06664）はこの問題を「Transitive Trust」として分析し、委任チェーン全体の権限追跡の必要性を指摘しています。

### Agent Impersonation（エージェントなりすまし）

Agent CardのURLを偽装し、正規エージェントになりすまして通信に参加する脅威です。A2AのAgent Card発見メカニズム（`/.well-known/agent.json`）は、DNSとTLSの信頼に依存しています。DNS汚染やTLS証明書の不正取得が成立すれば、攻撃者は正規エージェントとして振る舞えます。

### Data Exfiltration（データ窃取）

正規の通信チャネルを利用したデータ窃取です。マルチエージェント環境では、エージェント間でデータが流れることは正常な動作であるため、**正規通信と窃取の区別が困難**です。Greshake et al.（arXiv:2302.12173）は、間接プロンプトインジェクションとマルチエージェント環境の組み合わせがデータ窃取リスクを飛躍的に高めることを示しています。

### 対策の方向性

これらの脅威への対策として、以下の設計原則が求められます。

1. **暗号学的ID検証**: Agent Cardの自己申告ではなく、暗号署名によるID検証
2. **権限の非推移性**: 委任チェーンにおいて、元の権限スコープを超えないことの保証
3. **通信の暗号化と完全性検証**: mTLSによるエージェント間通信の保護
4. **行動ベースの信頼度評価**: 静的な信頼ではなく、継続的な信頼度スコアリング

### 信頼境界の設計パターン

マルチエージェントシステムにおける信頼境界は、以下のパターンで設計します。

**パターン1：Hub-and-Spoke（中央集権型）**

オーケストレーターが全通信を仲介し、エージェント間の直接通信を禁止します。信頼の管理が単純化される一方、オーケストレーターがSPOF（単一障害点）となるリスクがあります。

**パターン2：Mesh with Mutual Authentication（メッシュ型）**

各エージェントがmTLSで相互認証し、直接通信します。SPIFFE/SPIREと組み合わせることで、暗号学的に検証されたID同士の通信のみを許可できます。運用の複雑さが増しますが、耐障害性に優れます。

**パターン3：Zero Trust Mesh（ゼロトラスト型）**

メッシュ型に加え、すべての通信をポリシーエンジンで検証します。「信頼できるエージェント」という概念を排除し、各リクエストを個別に評価します。最もセキュアですが、レイテンシとコストのトレードオフがあります。

## ツール紹介

### Microsoft Agent Governance Toolkit

Microsoftが公開したマルチエージェントガバナンスのオープンソースツールキットです。以下の機能を提供します。

**暗号学的エージェントID**

Ed25519暗号鍵ペアによるエージェントIDの生成と、SPIFFE/SVIDとの統合をサポートします。各エージェントが暗号学的に検証可能なIDを持ち、Agent Cardの自己申告に依存しない認証を実現します。

**Trust Score（信頼度スコア）**

0〜1000のスケールでエージェントの信頼度を動的に評価します。通信履歴、ポリシー遵守状況、異常行動の有無を継続的にモニタリングし、スコアが閾値を下回った場合にアクセスを制限します。

```
エージェント登録 → 初期Trust Score: 500
  正常動作の継続  → スコア上昇（最大1000）
  ポリシー違反検出 → スコア低下
  スコア < 200     → 通信を遮断/人間承認を要求
```

**ポリシーエンジン**

OWASPの脅威カテゴリに対応したポリシー評価をサブミリ秒で実行します。Confused Deputy、Privilege Escalation、Agent Impersonationなど、本章で述べた脅威パターンに対するガードレールを宣言的に定義できます。

### A2A公式セキュリティガイド

A2Aリポジトリに含まれるセキュリティガイドラインです。Agent Cardでの認証方式宣言、TLS必須化、トークンスコープの制限など、プロトコルレベルでのセキュリティベストプラクティスを定義しています。ただし、前述の通り、ガイドラインの遵守は実装者に委ねられており、プロトコル自体が強制する仕組みは限定的です。

### 本章の脅威に対するツール選定の考え方

マルチエージェント通信のセキュリティは、プロトコル層（A2A/mTLS）、ID層（SPIFFE/Ed25519）、ガバナンス層（Trust Score/ポリシーエンジン）の多層で対処する必要があります。単一のツールで全てをカバーすることは困難であり、各層に適したツールを組み合わせる設計が現実的です。なお、Aigisはエージェント単体のアクション制御に特化しており、エージェント間通信プロトコルのセキュリティは対応範囲外です。

:::message
**2026年アップデート：OWASP ASI07とA2Aセキュリティの進展**

OWASP Agentic Top 10 2026では、**ASI07「Insecure Inter-Agent Communication」**が新設され、マルチエージェント通信の脆弱性が独立したリスクカテゴリとして認識されました。本章で解説した脅威モデル（Confused Deputy、Privilege Escalation Chain、Agent Impersonation、Data Exfiltration）が、OWASPの公式リスクリストに組み込まれた形です。

A2Aプロトコル自体も、Linux Foundation移管後に100社超のコミュニティで活発に開発が進んでいます。Gartnerは**2026年末までにエンタープライズアプリの40%がタスク固有のAIエージェントを含む**と予測しており、マルチエージェント環境のセキュリティ設計は「将来の課題」ではなく「今日の実装要件」になっています。

加えて、Google DeepMindの**CaMeL**（arXiv:2503.18813）が提案するCapabilityベースのデータフロー制御は、マルチエージェント環境にも応用可能です。エージェント間で受け渡されるデータにテイントラベルを付与し、非信頼データが制御フロー（他のエージェントへの指示等）に影響することを構造的に防止する設計パターンは、本章で述べたConfused Deputy問題への根本的な解決策となり得ます。
:::
