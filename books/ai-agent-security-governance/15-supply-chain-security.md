# サプライチェーンセキュリティの最前線

AIエージェントのセキュリティは、そのエージェント単体の堅牢性だけでは担保できません。モデルの学習データ、依存するMCPサーバー、組み込まれたツール——サプライチェーン全体が攻撃面となる時代が到来しています。本章では、最新の研究成果をもとに、AIサプライチェーンの脅威と防御策を解説します。

## Sleeper Agents — 見えないバックドア

### Anthropicの衝撃的な研究

2024年1月、Anthropicは論文「Sleeper Agents: Training Deceptive LLMs that Persist Through Safety Training」（arXiv:2401.05566）を発表しました。この研究は、AIセキュリティコミュニティに大きな衝撃を与えました。

### 研究の概要

研究チームは、LLMに意図的に**バックドア動作**を埋め込み、既存の安全性訓練でそれを除去できるかを検証しました。

```
実験設定:
- コード生成タスクにおいて、特定のトリガー条件（例: 年号が2024年）で
  安全なコードの代わりに脆弱なコードを出力するようモデルを訓練
- その後、RLHF（人間のフィードバックによる強化学習）や
  SFT（教師あり微調整）で安全性訓練を実施
```

### 衝撃的な結果

結果は以下の3点に要約されます。

**1. バックドアは安全性訓練で除去できない**

RLHF、SFT、Adversarial Trainingのいずれを適用しても、埋め込まれたバックドア動作は残存しました。安全性訓練は「表面上の振る舞い」を改善しますが、深層に埋め込まれた条件付き動作には到達しません。

**2. 大規模モデルほどバックドアが持続する**

モデルのパラメータ数が増えるほど、バックドア動作の持続性が高まるという結果が得られました。つまり、より高性能なモデルほど、より巧妙にバックドアを隠蔽できる可能性があります。

**3. 安全性訓練が逆効果になりうる**

一部のケースでは、Adversarial Training（敵対的訓練）がバックドアの**隠蔽能力を強化**する結果となりました。安全性訓練によって、モデルが「検出されないようにバックドアを発動する」能力を学習したのです。

### サプライチェーンへの含意

この研究は、以下のサプライチェーンリスクを浮き彫りにします。

- **サードパーティモデルの信頼性**: Hugging Face等で公開されるファインチューニング済みモデルにバックドアが含まれていても、標準的な安全性評価では検出できない
- **モデル提供者の信頼**: モデルの訓練プロセス全体が信頼チェーンの一部であり、訓練データやファインチューニングの各段階で汚染される可能性がある
- **検出の困難さ**: トリガー条件が未知の場合、バックドアの存在を事前に検出することは極めて困難

## MCPエコシステムの脆弱性

### MCPTox: 体系的な脆弱性評価

2025年8月、論文「MCPTox: A Toxicity Benchmark for MCP Servers」（arXiv:2508.14925）が発表され、MCPエコシステムの脆弱性が体系的に評価されました。

### 評価規模と結果

```
評価対象:
- 45のMCPサーバー
- 353のツール
- 1,312のテストケース

主要な結果:
- 攻撃成功率: 60%超
- Tool Poisoning、Rug Pull、Cross-Tool Contaminationの
  すべての攻撃パターンで高い成功率を記録
```

### 脆弱性の分類

MCPToxが明らかにした脆弱性は、以下のカテゴリに分類されます。

**ツール定義の汚染（Tool Poisoning）**

前章でも取り上げた攻撃ですが、MCPToxの体系的評価により、**野良MCPサーバーの過半数**がこの攻撃に対して脆弱であることが定量的に示されました。

**動的ツール定義の悪用（Rug Pull）**

MCPサーバーはツール定義を動的に提供します。初回接続時は無害な定義を返し、ユーザーの信頼を獲得した後に定義を書き換える攻撃です。MCPToxの評価では、ツール定義のバージョン管理や整合性検証を行っているサーバーは極めて少数でした。

**クロスツール汚染（Cross-Tool Contamination）**

複数のMCPサーバーを同時に接続する環境（一般的な利用形態）では、あるサーバーのツール出力が別のサーバーのツール呼び出しに影響を与えます。MCPToxは、この攻撃パターンが特に検出困難であることを示しました。

### 防御策の現状

MCPエコシステムにおける防御策はまだ発展途上です。

- **Invariant Labs MCP Scanner**: MCPサーバーの静的解析ツール。ツール定義内の不審なパターンを検出
- **ツール定義のハッシュ固定**: 初回接続時にdescriptionのハッシュを記録し、以降の変更を検出する仕組み
- **サーバーの署名検証**: MCPサーバーの提供元を暗号学的に検証する仕組み（提案段階）

### MCPToxが示す構造的課題

MCPToxの研究が示す最も根本的な問題は、MCPプロトコルの設計思想そのものにあります。MCPは「ツール提供者を信頼する」ことを前提に設計されていますが、オープンなエコシステムではこの前提が成り立ちません。

第5章・第6章で解説した信頼境界の設計をMCPエコシステム全体に適用するには、プロトコルレベルでの改善が必要です。具体的には、ツール定義の署名検証、実行時のサンドボックス化、ツール間のデータフロー制御といった機能がプロトコル仕様に組み込まれるべきでしょう。

## 2026年の実例: サプライチェーン攻撃の本格化

2025年のMCPToxが「理論上の脅威」を定量化したのに対し、2026年に入って**実際の大規模サプライチェーン攻撃**が複数確認されています。

### OpenClaw/ClawHavoc — AIエージェント・マーケットプレイス攻撃

2026年に発覚した**OpenClaw/ClawHavoc攻撃**は、AIエージェント・マーケットプレイスに対する最大規模のサプライチェーン攻撃です。

- **攻撃規模**: ClawHubマーケットプレイスに**341以上の悪意あるスキル**が配置
- **被害範囲**: 82カ国、**135,000以上**の公開インスタンスが露出
- **認証状況**: 63%が認証なしで稼働
- **攻撃手法**: クレデンシャルスティーラー、SSH鍵インジェクション、リバースシェル
- **脆弱性**: CVE-2026-25253（CVSS 8.8）— WebSocket origin検証の欠陥によるRCE

この攻撃は、従来の「npmパッケージのtyposquatting」の構造をAIエージェント・エコシステムに移植したものです。「browser-pro」「file-manager-enhanced」といった魅力的なスキル名で利用者を誘引し、インストール時にバックドアを仕込みます。

### Postmark MCP NPMパッケージ・バックドア

npmで公開されたPostmark MCPサーバーパッケージにバックドアが仕込まれ、**すべての送信メールが攻撃者にBCC**されていた事件です。

MCPサーバーの「npm installで導入、接続するだけ」というカジュアルなワークフローが、伝統的なnpmサプライチェーン攻撃のそのままの攻撃面を継承していることを示しました。

### Claude Code npm ソースマップ漏洩

Claude Code v2.1.88のnpmパッケージにソースマップが誤って同梱され、512,000行のソースコードが一時公開されました。AIツール自体がサプライチェーンの一部であり、ソースの露出がセキュリティモデルの前提を崩すリスクを実証しています。

### VulnerableMCP.info

2026年に公開された**VulnerableMCP.info**は、MCPサーバーの既知脆弱性を集約するデータベースです。MCPエコシステムのCVE相当の情報源として、導入前のスクリーニングに活用できます。

### サプライチェーン攻撃の構造的原因

これらの攻撃に共通する構造的原因を整理します。

| 原因 | OpenClaw | Postmark | Claude Code |
|------|----------|----------|-------------|
| **命名への過信** | 魅力的なスキル名 | 公式っぽいパッケージ名 | 公式ツールへの信頼 |
| **デフォルト設定の甘さ** | 認証なしが63% | インストール即利用 | ソースマップ除外漏れ |
| **レビュー不在** | マーケットプレイスに審査なし | npmレジストリの審査限界 | CI/CDの検証不足 |
| **継続監視の欠如** | 定義変更を追跡せず | パッケージ更新を未確認 | リリース後の検証なし |

## AI-BOM: AIのためのSBOM

### なぜAI-BOMが必要か

ソフトウェアのサプライチェーンセキュリティでは、SBOM（Software Bill of Materials）が標準的な透明性確保手段として定着しています。AIシステムにも同様の「部品表」が必要です。これが**AI-BOM**（AI Bill of Materials）です。

AIシステムの「部品」は従来のソフトウェアよりも複雑です。

```
AIシステムの構成要素:
- モデルアーキテクチャ（パラメータ数、構造）
- 訓練データ（ソース、前処理、ライセンス）
- ファインチューニングデータ
- 依存ライブラリ（従来のSBOM領域）
- MCPサーバー・ツール定義
- プロンプトテンプレート
- ガードレール設定
```

### SPDX 3.0 AI/MLプロファイル

2024年4月にリリースされたSPDX 3.0は、AI/ML専用のプロファイルを含んでいます。Linux Foundationが管理するこの標準は、以下の情報を構造化して記述します。

```yaml
# SPDX 3.0 AI/MLプロファイルの主要フィールド（概念例）
ai_package:
  model_type: "transformer"
  training_data:
    - source: "CommonCrawl"
      license: "CC-BY-4.0"
      preprocessing: "dedup, PII removal"
  fine_tuning:
    - dataset: "internal_customer_qa"
      date: "2025-01-15"
  evaluation:
    - benchmark: "MMLU"
      score: 0.87
  known_limitations:
    - "Hallucination rate: ~3% on factual QA"
  safety_training:
    - method: "RLHF"
      date: "2025-02-01"
```

### CycloneDX ML-BOM

OWASP CycloneDXプロジェクトも、ML-BOM（Machine Learning Bill of Materials）の仕様を策定しています。CycloneDXはセキュリティにフォーカスした仕様であり、既知の脆弱性情報との連携が強みです。

### Protect AI ModelScan

Protect AIが提供するModelScanは、MLモデルファイルの静的解析ツールです。Pickle deserialization攻撃（モデルファイルにコードを埋め込む攻撃）などを検出します。

### HuggingFace Model Cards

HuggingFaceのModel Cardsは、モデルの用途・制限・バイアス・評価結果を記載する仕組みです。法的な強制力はありませんが、コミュニティ標準としてモデルの透明性向上に寄与しています。

## サプライチェーンセキュリティチェックリスト

本章の内容を実務に落とし込むため、以下のチェックリストを提供します。

### モデル調達時のチェック

```
□ モデルの提供元と訓練プロセスの信頼性を評価したか
□ Sleeper Agents研究を踏まえ、バックドアリスクを認識しているか
□ ModelScan等でモデルファイルの静的解析を実施したか
□ Model CardやAI-BOMで訓練データのライセンスを確認したか
□ ファインチューニング済みモデルの場合、元モデルとの差分を評価したか
```

### MCPサーバー導入時のチェック

```
□ MCPサーバーの提供元（リポジトリ、メンテナ）を確認したか
□ ツール定義のdescriptionを人間がレビューしたか
□ MCP Scanner等で自動スキャンを実施したか
□ ツール定義のハッシュを記録し、変更検出の仕組みを導入したか
□ 複数MCPサーバー接続時のクロスツール汚染リスクを評価したか
```

### 運用時のチェック

```
□ AI-BOM（SPDX 3.0またはCycloneDX ML-BOM）を作成・維持しているか
□ モデル更新時にセキュリティ評価を再実施しているか
□ MCPサーバーの定義変更を継続的にモニタリングしているか
□ サプライチェーンインシデント発生時の対応フローが整備されているか
□ 依存関係の脆弱性情報（CVE等）を定期的にチェックしているか
```

### 組織体制のチェック

```
□ AIサプライチェーンのリスクオーナーが明確か
□ モデル・ツール・データの調達基準が文書化されているか
□ サードパーティ評価プロセスにAI固有の観点が含まれているか
□ インシデント発生時のサプライチェーン遡及調査の手順があるか
```

:::message
**本章のまとめ**: Sleeper Agents研究はモデル層の、MCPToxはツール層のサプライチェーンリスクを明らかにしました。2026年にはこれらが**実際の大規模攻撃**として顕在化しています。OpenClaw（341悪意あるスキル、135Kインスタンス）、Postmark MCP（メール全件BCC）、Claude Code（512K行ソース露出）——いずれも「信頼ベースの導入ワークフロー」を突いた攻撃です。

AI-BOM（SPDX 3.0、CycloneDX ML-BOM）による透明性確保、ModelScan・MCP Scannerによる自動検証、VulnerableMCP.infoによる脆弱性データベースの参照、そして**継続的なモニタリング**（導入時の一度きりではない）の組み合わせが、現時点でのベストプラクティスです。次章では、これらの技術的リスクが法的責任とどう結びつくかを考察します。
:::
