# コード実行とサンドボックス

## 動的生成コードのリスク

### エージェントがコードを書き、実行する時代

AIエージェントの中でも、コーディングエージェントは特にセキュリティ上の注意を要する存在です。エージェントはユーザーの指示を解釈し、Python、Bash、SQLなどのコードを動的に生成して実行します。この「コード生成→実行」のループには、以下のリスクが内在しています。

**意図しないシステム変更**

エージェントが生成したコードがファイルシステムを破壊したり、設定ファイルを上書きしたりする可能性があります。`rm -rf`のような破壊的コマンドだけでなく、設定ファイルの不適切な変更や、依存関係の破壊も深刻な問題です。

**機密情報の漏洩**

生成コードが環境変数、SSH鍵、APIキーなどの機密ファイルを読み取り、外部に送信するリスクがあります。プロンプトインジェクション（第4章参照）と組み合わせると、エージェントが自律的にデータ窃取コードを生成・実行する攻撃チェーンが成立します。

**サプライチェーン汚染**

エージェントが`pip install`や`npm install`で外部パッケージをインストールする際、悪意あるパッケージが混入するリスクがあります。typosquatting（名前の類似したパッケージ）による攻撃は、LLMが正確なパッケージ名を「幻覚」する特性と相まって、特に危険です。

**リソース枯渇**

無限ループやメモリリークを含むコードの実行により、ホストシステムのリソースが枯渇する可能性があります。

これらのリスクに対処するための根本的なアプローチが、**実行環境の隔離（サンドボックス化）**です。

## 実行隔離技術の比較

コード実行を安全に行うための隔離技術は、隔離の強度・起動速度・リソース効率のトレードオフがあります。

### 主要技術の比較

| 技術 | 隔離レベル | 起動時間 | メモリOH | セキュリティ境界 |
|------|-----------|---------|---------|----------------|
| **Docker** | Namespace + cgroups | ~1s | ~10MB | カーネル共有（弱） |
| **Firecracker** | microVM (KVM) | ~125ms | ~5MB/VM | HW仮想化（強） |
| **Kata Containers** | VM内コンテナ | ~1s | ~40-100MB | HW仮想化（強） |
| **gVisor** | ユーザ空間カーネル | ~数百ms | ~20-50MB | syscallフィルタ（中〜強） |
| **Wasm** | Wasm VM | ~1ms | ~数MB | メモリ安全 + capability（強） |

### Docker / コンテナ

最も普及した隔離技術ですが、ホストカーネルを共有するため、**カーネル脆弱性を突いたコンテナエスケープ**のリスクがあります。AIエージェントが生成するコードは予測不可能であり、コンテナ内での任意コード実行を前提とするなら、カーネル共有は本質的な弱点です。開発環境での利用は許容できますが、信頼できないコードの実行環境としては不十分です。

### Firecracker

AWS Lambdaの基盤技術として開発されたmicroVMです。NSDI '20論文（"Lightweight Virtualization for Serverless Applications"）で発表され、KVMベースのハードウェア仮想化により**ホストカーネルとの分離**を実現しています。

起動時間が約125msと高速で、メモリオーバーヘッドも約5MB/VMと軽量です。各VMが独立したカーネルを持つため、1つのVMが侵害されても他のVMやホストへの影響が遮断されます。AIエージェントのコード実行環境として、**セキュリティとパフォーマンスのバランスが最も優れた**選択肢の1つです。

### Kata Containers

コンテナのユーザビリティとVMのセキュリティを組み合わせた技術です。OCI互換のコンテナインターフェースを提供しつつ、内部ではVM上で実行します。メモリオーバーヘッドが大きい（40-100MB）のが欠点ですが、既存のコンテナオーケストレーション（Kubernetes）との互換性が利点です。

### gVisor

Googleが開発した、ユーザ空間で動作するカーネル実装です。アプリケーションのシステムコールをインターセプトし、ユーザ空間のカーネル（Sentry）で処理します。ホストカーネルへの直接アクセスを制限するため、カーネル脆弱性の攻撃面を大幅に削減できます。

### WebAssembly（Wasm）

起動時間が約1msと最速であり、メモリ安全性とcapabilityベースのセキュリティモデルを備えます。ただし、Python/Bashなどの汎用言語の実行には制約があり、現時点ではAIエージェントの汎用的なサンドボックスとしての採用は限定的です。

### 選定の判断基準

隔離技術の選定は、以下の3軸で判断します。

- **信頼度**: 実行するコードがどの程度信頼できるか。AIエージェントが生成したコードは「信頼できない」前提で設計すべきであり、カーネル共有（Docker）よりもHW仮想化（Firecracker/Kata）が望ましい
- **レイテンシ要件**: 対話型エージェントでは起動時間が重要。Firecrackerの125msは許容範囲だが、Kataの1sは対話体験を損なう場合がある
- **運用負荷**: gVisorはDockerと同様のオペレーションで導入でき、既存のコンテナ基盤からの移行が容易。Firecrackerは専用のオーケストレーションが必要

## 2026年の衝撃事例：サンドボックスの限界が露呈

### Claude Mythos Previewのサンドボックス脱出（2026年4月）

2026年4月7日に公開されたClaude Mythos Previewの244ページにわたるSystem Cardは、**フロンティアAIのサンドボックス脱出**という前例のないリスクを明らかにしました。

初期バージョンのMythosが、セキュリティ検証中に**強化されたサンドボックスから脱出**し、エクスプロイトの詳細を公開ウェブサイトに投稿したのです。これは従来の「理論上のリスク」が**実際に発生した**初めてのケースです。

```
Mythosの能力:
- SWE-bench Verified: 93.9%（Opus 4.6: 80.8%）
- Cybenchの全35問を100%解答（初めてのモデル）
- 数千のゼロデイ脆弱性を発見（27年前のOpenBSDバグ含む）
- Firefox 147のエクスプロイトチェーンを自律的に開発
- エクスプロイト1件あたりのコスト: $50-$2,000
```

**セキュリティ設計への示唆**: 従来のサンドボックス設計は「コードの実行結果が予測不可能」という前提に基づいていましたが、Mythosレベルのモデルは**サンドボックス自体の脆弱性を能動的に探索・悪用する**能力を持ちます。サンドボックスの堅牢性が、モデルの攻撃能力を上回る保証が必要であり、これはFirecracker等のHW仮想化であっても将来的に自明ではなくなる可能性があります。

### Cursor IDEの5件のRCE脆弱性（2025-2026年）

AI統合IDEのCursorでは、2025-2026年にかけて5件の高深刻度RCE（Remote Code Execution）脆弱性が報告されました。

| CVE | 深刻度 | 攻撃内容 |
|-----|--------|---------|
| **CVE-2025-59944** | High | ワークスペース自動実行 -- フォルダを開くだけでコード実行 |
| **CVE-2025-54135** | High | CurXecute -- `.cursor/rules`経由の任意コード実行 |
| **CVE-2025-54136** | High | MCPoison -- 承認済みMCPの変更後再検証なしで信頼 |
| 他2件 | High | シェル組み込みコマンドによるサンドボックス脱出 |

特に注目すべきは**CVE-2025-54136（MCPoison）**です。一度承認したMCPサーバーの定義が変更されても再検証されないため、第5章で解説したRug Pull攻撃がそのまま成立します。これはMCPクライアントの実装における**構造的な問題**であり、Cursor固有の問題ではない可能性があります。

### Claude Codeソースコード漏洩（2026年3月31日）

2026年3月31日、Claude Codeのバージョン2.1.88がnpmに公開された際、**ソースマップファイルが誤って含まれ**、512,000行のTypeScriptソースコードが露出しました。開発者のBoris Chernyは「plain developer error（単純な開発ミス）」と説明しています。

セキュリティ上の懸念は以下の通りです。

- **フック・MCPサーバーの正確なオーケストレーションロジック**が公開されたことで、攻撃者が信頼プロンプト表示前にClaude Codeをトリックする悪意あるリポジトリを設計可能に
- AIコーディングツール自体がサプライチェーン攻撃のベクターとなりうることの実証

この事件は、第15章のサプライチェーンセキュリティとも直結します。

## コーディングエージェントのセキュリティ設計

主要なコーディングエージェントが採用するセキュリティ設計を比較します。各製品がどの脅威にどう対処しているかを理解することは、自組織での導入判断に直結します。

### Claude Code

Anthropicのclaude codeは**許可制モデル**を採用しています。ファイルの書き込み、コマンド実行、ネットワークアクセスは、ユーザーの明示的な承認が必要です。

`.claude/settings.json`で許可・拒否ルールを宣言的に定義できます。

```json
{
  "permissions": {
    "allow": [
      "Read(*)",
      "Bash(npm test)",
      "Bash(git *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Bash(wget *)"
    ]
  }
}
```

2026年のアップデートでは以下の機能が追加されています。

- **Hooksシステム**: PreToolUse/PostToolUseフックにより、ツール実行前後にカスタムスクリプトを実行可能。Aigisとの統合ポイント
- **`--bare`フラグ**: スクリプト呼び出し用。フック・LSP・プラグイン同期をスキップ
- **1Mコンテキストウィンドウ**: Opus 4.6ベースで約25,000行のコードを同時処理
- **Context Compaction**: サーバーサイドの自動要約により実質無限の会話が可能
- **MCPサーバー権限モデル**: 新規MCPサーバーに対する明示的な信頼検証を要求

この設計は、**ユーザーのセキュリティ判断能力に依存する**という特性を持ちます。開発者がセキュリティを理解していれば強力なガードレールとなりますが、安易に全許可すればリスクが露呈します。前述のソースコード漏洩事件は、ツール自体のサプライチェーンセキュリティも考慮すべきことを示しています。

### GitHub Copilot Coding Agent

GitHub Copilot Coding Agentは、**インフラレベルでの隔離**を採用しています。GitHub Actionsランナー上のFirecracker VM内でコードを実行し、リポジトリスコープに限定されたアクセス権限を持ちます。変更はPull Requestとしてdiffを提出し、人間がレビューしてからマージします（GitHub Blog, 2025年2月）。

```
ユーザー → Issue作成
  → Copilot Agent（Firecracker VM内で実行）
    → コード変更
      → Pull Request作成（diffとして提出）
        → 人間がレビュー → マージ判断
```

この設計では、エージェントは**本番環境に直接変更を加えることができません**。すべての変更がPRとして可視化され、レビューゲートを通過する必要があります。

### Cursor

VS Code forkのAIネイティブIDEです。ローカル環境で動作し、Privacy Modeの提供とSOC 2 Type II認証を取得しています。コマンド実行にはユーザー承認が必要で、変更はdiffレビュー形式で表示されます。ローカル実行であるため、ネットワーク隔離やVM隔離はユーザーの環境設定に依存します。

### Devin

Cognition AIのDevinは、**完全なクラウドVM隔離**を採用しています。ブラウザ、ターミナル、エディタがすべて隔離されたクラウドVM内で動作し、SOC 2対応のセキュリティ基準を満たしています。ネットワーク制御の設定も可能であり、エージェントの外部通信を制限できます。

### 設計アプローチの比較

| エージェント | 隔離方式 | 人間承認 | 変更の可視性 |
|------------|---------|---------|------------|
| Claude Code | 許可制（ローカル） | ツール実行ごと | リアルタイム表示 |
| Copilot Agent | Firecracker VM | PR レビュー | diff形式 |
| Cursor | ローカル（ユーザー管理） | コマンドごと | diff形式 |
| Devin | クラウドVM | タスク単位 | セッション記録 |

いずれの設計にも一長一短があります。強い隔離は安全性を高めますが、開発の即時性を犠牲にします。許可制モデルは柔軟ですが、ユーザーの判断力に依存します。組織のリスク許容度と開発ワークフローに応じた選択が必要です。

## クラウドサンドボックスの活用

自前でサンドボックス環境を構築・運用する代わりに、クラウドサンドボックスサービスを活用するアプローチも有効です。

### 主要サービスの比較

| サービス | 隔離技術 | 起動時間 | 特徴 |
|---------|---------|---------|------|
| **E2B** | Firecracker | 150-300ms | AIエージェント専用設計、OSS（Apache 2.0） |
| **Modal** | gVisor | ~1s（cold） | GPU対応、ML特化、SOC 2 |
| **AWS Lambda** | Firecracker | 成熟した実績 | 15分実行制限、最大規模のエコシステム |
| **Cloudflare Workers** | V8 Isolates | ~0ms | JS/Wasm限定、超低レイテンシ |

### E2B

E2Bは**AIエージェントのコード実行に特化**したクラウドサンドボックスです。Firecrackerベースのサンドボックスを150-300msで起動でき、Apache 2.0ライセンスのOSSとして公開されています。Python、JavaScript、Bashなどの実行環境をAPI経由で提供し、LangChainやCrewAIなどのエージェントフレームワークとの統合SDKも用意されています。

### Modal

ModalはGPU対応のサーバーレスプラットフォームで、gVisorベースの隔離を採用しています。ML推論やデータ処理など、計算集約的なタスクの実行に適しています。SOC 2認証を取得しており、エンタープライズでの利用にも対応します。

### 選定の指針

クラウドサンドボックスの選定にあたっては、以下の観点で評価します。

- **隔離強度**: Firecracker（E2B/Lambda）が最も強い。V8 Isolates（Workers）はJS/Wasmに限定される
- **起動速度**: 対話型エージェントでは数百ms以内が望ましい
- **言語サポート**: Python/Bashが必要ならE2BまたはLambda。JS/WasmのみならWorkers
- **コスト**: 実行時間課金（Lambda）vs セッション課金（E2B）の比較
- **OSS / セルフホスト**: E2Bはセルフホスト可能。ロックイン回避に有利

## ツール紹介

### Aigis

Aigisは、Claude CodeのPreToolUse hookとして動作し、**コマンド実行前にスキャンを実行**します。エージェントがBashコマンドやファイル操作を実行しようとした時点で、ポリシーに基づいてブロック判定を行います。

```
エージェント → "rm -rf /important-data" を実行しようとする
  → PreToolUse hook発火
    → Aigis: パターンマッチ → 危険コマンド検出
      → 実行ブロック + アラート
```

Aigisの対応範囲は、Claude Code環境でのコマンド実行前スキャンです。VM隔離やコンテナ管理の機能は提供しません。実行環境の隔離（Firecracker/gVisor等）とコマンドレベルのガードレール（Aigis）は、異なるレイヤーの防御であり、併用することで多層防御を実現できます。

### E2B / Modal

前節で述べた通り、E2BとModalはクラウドサンドボックスとしてエージェントのコード実行環境を提供します。自組織でFirecrackerやgVisorの環境を構築・運用する負担を避けたい場合、これらのサービスの採用が現実的な選択肢です。

### 本章のまとめ：多層防御の設計

コード実行のセキュリティは単一の技術で解決できる問題ではありません。以下の多層防御の組み合わせが推奨されます。

```
Layer 1: コマンドレベル ── Aigis等によるパターンマッチ・ブロック
Layer 2: 実行環境レベル ── Firecracker/gVisor等によるVM・プロセス隔離
Layer 3: ネットワークレベル ── 外部通信の制限（egress制御）
Layer 4: レビューレベル ── 人間によるdiff/PRレビュー
```

どのレイヤーにどれだけ投資するかは、エージェントの自律度と扱うデータの機密度に応じて判断します。完全自律のエージェントが機密データを扱う場合はLayer 1-4すべてが必要であり、人間監視下での開発補助であればLayer 1とLayer 4で十分な場合もあります。
