# 監査・オブザーバビリティ

## エージェントオブザーバビリティの5要素

従来のソフトウェアオブザーバビリティが「メトリクス・ログ・トレース」の3本柱で構成されるのに対し、AIエージェントのオブザーバビリティは5つの要素を必要とします。エージェントは非決定論的に動作し、外部ツールと連携し、自律的に判断を下すため、従来の3本柱だけでは「何が起きたか」を十分に把握できません。

### 1. 推論トレース（Reasoning Trace）

エージェントがどのような思考過程を経て行動を決定したかの記録です。Chain-of-Thought（CoT）の各ステップ、ツール呼び出しの判断理由、代替案の検討と棄却の過程が含まれます。

推論トレースは、インシデント発生時の根本原因分析に不可欠です。「なぜエージェントはこのアクションを選択したのか」を事後に検証するには、推論の過程が記録されている必要があります。

### 2. ツール呼び出しログ（Tool Invocation Log）

エージェントが呼び出したすべてのツール・API・外部サービスの記録です。入力パラメータ、出力結果、レイテンシ、エラー情報を含みます。

```json
{
  "timestamp": "2026-04-01T10:23:45Z",
  "agent_id": "data-analyst-01",
  "tool": "database_query",
  "input": {
    "query": "SELECT count(*) FROM orders WHERE date > '2026-03-01'",
    "database": "analytics-readonly"
  },
  "output": {
    "rows": 1,
    "result": [{"count": 15234}]
  },
  "latency_ms": 45,
  "policy_evaluation": {
    "rule": "allow-readonly-analytics",
    "result": "allow"
  }
}
```

### 3. データアクセスログ（Data Access Log）

エージェントがアクセスしたデータソース、読み取ったファイル、送信したデータの記録です。PII（個人識別情報）やPHI（保護された医療情報）へのアクセスは特に重要であり、GDPR・HIPAA等の規制においてアクセスログの保持が義務付けられています。

### 4. パフォーマンスメトリクス

トークン消費量、レイテンシ、タスク完了率、リトライ回数、コストといった定量的指標です。異常値の検出（通常の10倍のトークン消費など）は、プロンプトインジェクション攻撃やエージェントのスタック（無限ループ）を示唆する可能性があります。

### 5. ガードレール違反ログ（Guardrail Violation Log）

ポリシーエンジンがブロックした行動、HITLで却下されたアクション、安全フィルタが発動した事象の記録です。違反の傾向分析により、ポリシーの過不足やエージェント設計の問題を発見できます。

これら5つの要素を統合的に収集・可視化・分析することが、エージェントオブザーバビリティの目標です。

## OpenTelemetry GenAI

### セマンティック規約

OpenTelemetry（OTel）はCNCFのGraduatedプロジェクトであり、分散トレーシング・メトリクス・ログの業界標準です。2024年から、GenAI（生成AI）に特化したセマンティック規約の策定が進められています。現在はExperimentalステータスですが、急速に成熟しています。

GenAIセマンティック規約は、以下の属性を標準化しています。

| 属性 | 説明 | 例 |
|---|---|---|
| `gen_ai.system` | AIプロバイダの識別子 | `openai`, `anthropic` |
| `gen_ai.request.model` | 使用モデル名 | `claude-sonnet-4-20250514` |
| `gen_ai.request.max_tokens` | 最大トークン数 | `4096` |
| `gen_ai.response.finish_reasons` | 完了理由 | `["stop"]` |
| `gen_ai.usage.input_tokens` | 入力トークン数 | `1523` |
| `gen_ai.usage.output_tokens` | 出力トークン数 | `847` |

### OpenLLMetry

OpenLLMetry（Traceloop社）は、OpenTelemetryのGenAI計装を簡単に実装するためのオープンソースライブラリです（https://github.com/traceloop/openllmetry）。Python、TypeScript、Java等の主要言語に対応し、LangChain、LlamaIndex、CrewAI等の主要フレームワークとの自動計装を提供します。

```python
from traceloop.sdk import Traceloop

# 初期化のみで自動計装が有効化
Traceloop.init(
    app_name="my-agent",
    disable_batch=False,
    api_endpoint="http://localhost:4318",  # OTel Collector
)
```

OpenLLMetryの価値は、既存のOpenTelemetryエコシステム（Jaeger、Grafana Tempo、Datadog等）にGenAIのトレースデータをシームレスに統合できる点にあります。新たなオブザーバビリティ基盤を構築する必要がなく、既存の投資を活かせます。

### エージェント固有の計装課題

標準的なGenAIセマンティック規約は単一のLLM呼び出しを対象としていますが、エージェントは1つのタスクで複数のLLM呼び出しとツール実行を連鎖させます。この「エージェントスパン」——タスク開始からタスク完了までの全体をトレースとして追跡する仕組みは、まだ標準化が進行中です。

現時点では、エージェントフレームワーク（LangChain、LangGraph等）が独自のトレース構造を提供しており、OpenTelemetryへのブリッジを経由して標準エコシステムに接続する形が主流です。

## プラットフォーム比較

GenAIオブザーバビリティのプラットフォームは急速に増加しています。主要なプラットフォームの特徴と選定基準を整理します。

### LangSmith

LangChain社が提供するSaaSプラットフォームで、LangChain/LangGraphとのネイティブ統合が最大の強みです。料金は$39/seat/月〜のチーム向けプランから提供されています。

**強み**: LangChainエコシステムとのシームレスな統合、プロンプトのバージョン管理、評価（Eval）機能の統合。
**弱み**: LangChainへのベンダーロックイン、オンプレミス対応なし。

### LangFuse

オープンソースのGenAIオブザーバビリティプラットフォームで、セルフホストが可能です。LangChainに限定されず、OpenAI、Anthropic、LlamaIndex等の多様なフレームワークに対応しています。

**強み**: OSS（セルフホスト可）、マルチフレームワーク対応、コスト管理機能。
**弱み**: セルフホスト時の運用負荷、エンタープライズサポートの成熟度。

データ主権やコスト制約の観点から、LangFuseのセルフホストモデルは日本企業にとって魅力的な選択肢です。

### Helicone

プロキシベースのアプローチを採用しており、アプリケーションコードの変更なしにGenAI呼び出しの監視を開始できます。APIエンドポイントをHeliconeのプロキシに向けるだけで計装が完了します。

**強み**: 導入の容易さ（コード変更不要）、リアルタイムダッシュボード。
**弱み**: プロキシ経由のレイテンシ増加（通常は数ミリ秒程度）、エージェントの内部推論トレースは取得不可。

### Braintrust

評価（Eval）に特化したプラットフォームで、エージェントの出力品質を定量的にスコアリングする機能が充実しています。CI/CDパイプラインに組み込み、モデル変更やプロンプト変更の前後で品質回帰を検出する用途に適しています。

### エンタープライズ統合: Datadog / Dynatrace

既存のAPM（Application Performance Monitoring）基盤を持つ組織にとって、Datadog AI Observability（2024年GA）やDynatraceのAI監視機能は自然な選択肢です。

Datadogは既存のインフラメトリクス・APMトレース・ログと統合されたダッシュボードでGenAIの状態を監視でき、アラート設定も既存のルールエンジンを流用できます。Dynatraceも同様に、既存のAIOps機能とGenAI監視を統合しています。

エンタープライズ規模の組織では、GenAI専用ツール（LangSmith/LangFuse等）を開発チーム向けに採用し、Datadog/Dynatraceを全社監視基盤として位置付ける二層構造が現実的です。

### 選定フローチャート

```
LangChain/LangGraph利用？ → Yes → LangSmith
                           → No ↓
セルフホスト必須？ → Yes → LangFuse
                    → No ↓
コード変更を最小化？ → Yes → Helicone
                      → No ↓
Eval/品質管理重視？ → Yes → Braintrust
                     → No ↓
既存APM基盤あり？ → Yes → Datadog / Dynatrace
```

## コンプライアンスレポートの自動生成

オブザーバビリティデータの蓄積は、それ自体が目的ではなく、**コンプライアンスレポートの自動生成**が最終的な価値です。規制当局への報告、内部監査への対応、外部監査への証跡提供——これらを手動で作成するのは非現実的であり、オブザーバビリティデータから自動生成する仕組みが求められます。

### レポートに含めるべき項目

コンプライアンスレポートは、最低限以下の項目を含むべきです。

1. **エージェント稼働概要**: 対象期間中のエージェント数、タスク実行数、ツール呼び出し数
2. **ガードレール違反サマリ**: 違反の種類別件数、傾向、対処状況
3. **HITL実績**: 承認/却下の件数、承認率、平均承認時間
4. **データアクセスサマリ**: PII/PHIアクセス件数、アクセス先の分布
5. **インシデント対応記録**: 発生したインシデントと対応状況
6. **規制要件マッピング**: 各規制要件に対する充足状況

### 規制要件マッピングの自動化

複数の規制フレームワーク（OWASP、SOC 2、GDPR、日本AI事業者ガイドライン等）にまたがる要件を、オブザーバビリティデータから自動的に充足確認する仕組みは、コンプライアンスコストを大幅に削減します。

たとえば、GDPR第30条のデータ処理記録義務は、データアクセスログから自動的に充足を証明できます。SOC 2のCC6.1（論理的アクセス制御）は、ポリシーエンジンの評価ログから充足を示せます。

## ツール紹介

### Aigis — 3層アクティビティストリーム

Aigisは3層構造のアクティビティストリームでエージェントの行動を記録します。

**Localストリーム**: 個々のエージェントセッション内でのツール呼び出し、ポリシー評価結果、ブロックされたアクションをリアルタイムで記録します。開発者がデバッグや動作確認に使用します。

**Globalストリーム**: 組織全体のエージェント活動を集約し、ダッシュボード形式で可視化します。管理者がエージェント群の全体的な健全性を把握するために使用します。

**Alertsストリーム**: ガードレール違反、異常行動、ポリシー違反をリアルタイムで通知します。Escalation Triggerと連携し、HITLフローの起点となります。

データはCSVエクスポートに対応しており、既存のSIEMやBIツールとの統合が可能です。

Aigisのコンプライアンスレポート機能は、**24の規制要件**へのマッピングを提供します。対応する規制フレームワークはOWASP Agentic Top 10、SOC 2、GDPR、日本のAI事業者ガイドライン等です。各要件に対して、Aigisのどの機能が充足に寄与するかを自動的にマッピングし、レポートとして出力します。

ただし、Aigisのコンプライアンスレポートはガードレール・ポリシー評価を中心としたものであり、推論トレースやパフォーマンスメトリクスの詳細分析には前述のOpenTelemetryベースのプラットフォーム（LangFuse、Datadog等）との組み合わせが推奨されます。

### 推奨構成

エンタープライズ環境での推奨構成は以下のとおりです。

```
エージェント
  ├→ Aigis（ポリシー評価・ガードレール違反・コンプライアンスレポート）
  ├→ OpenLLMetry（OTel計装 → Collector → バックエンド）
  │    └→ LangFuse or Datadog（トレース可視化・メトリクス分析）
  └→ SIEM（セキュリティイベント統合）
```

この構成により、ポリシーガバナンス（Aigis）、技術的オブザーバビリティ（OTel + プラットフォーム）、セキュリティ監視（SIEM）の3層がカバーされます。

## まとめ

エージェントオブザーバビリティは、「何かが起きた後に調べる」ための仕組みではなく、「何かが起きる前に検知する」ためのプロアクティブな基盤です。5つの要素を網羅的に収集し、OpenTelemetryで標準化し、コンプライアンスレポートの自動生成につなげる。この一連のパイプラインが、規制対応と運用効率の両立を可能にします。
