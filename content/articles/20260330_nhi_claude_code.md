## はじめに——「AIエージェントのIDを管理しているか？」

チームでClaude Codeを使い始めたとき、ほとんどの組織が最初に決めることは「どのANTHROPIC_API_KEYを使うか」だ。そして気づけば、複数の開発者が同じキーを.envファイルに貼り付けて使っている。

これは認証情報の管理問題ではない。**アイデンティティ管理の問題だ。**

Claude Codeセッションは、ファイルシステムを読み書きし、シェルコマンドを実行し、外部APIを呼び出す。人間の開発者と同等かそれ以上の権限を持ちながら、アカウントも、ロールも、監査ログの紐付けもない存在として動いている。セキュリティの文脈ではこれを **NHI（Non-Human Identity）** と呼ぶ。

本記事はこの問いから始まる。あなたのチームで動いているClaude Codeは、誰として、どんな権限で、何にアクセスしているか——把握できているか。

---

## NHIとは何か

NHI（Non-Human Identity）は、人間ユーザーではなく、システム・アプリケーション・サービスが持つデジタルアイデンティティの総称だ。

セキュリティの世界ではNHIは新しい概念ではない。CIパイプラインが持つサービスアカウント、アプリケーションが外部APIを呼ぶためのAPIキー、マイクロサービス間の相互TLS証明書——これらはすべてNHIの古典的な例だ。

| カテゴリ | 具体例 | 主な管理課題 |
|---------|-------|------------|
| サービスアカウント | GCPサービスアカウント、AWSロール | 過剰な権限、退職者アカウントの残留 |
| APIキー・トークン | GitHub Token、Stripe APIキー | 平文での.envへの保存、ローテーション漏れ |
| 証明書・シークレット | SSL証明書、mTLS証明書 | 有効期限切れ、発行管理の分散 |
| OAuthクライアント | サードパーティ連携アプリ | スコープの肥大化、失効管理の欠如 |
| **AIエージェントセッション** | **Claude Codeセッション、MCPサーバー接続** | **権限定義なし、監査ログなし** |

2025年以降、最後の行——AIエージェントセッション——が急速に増加している。そしてこれが最もガバナンスが遅れているカテゴリだ。

## なぜ今NHIが注目されているか

### 侵害の68%がNHIの不正利用から

Verizon DBIR 2025は衝撃的な数字を報告している。企業への侵害のうち68%が、NHIに分類される認証情報の不正利用から発生しており、この割合は人間ユーザーの認証情報悪用を初めて超えた。

背景には単純な算数がある。**CyberArkの調査によると、企業が管理するNHIの数は人間アカウントの平均45倍に達している。** しかし管理の成熟度は逆転している。人間アカウントにはMFA・パスワードポリシー・定期的なアクセスレビューが適用されるが、NHIの大半は「作成して終わり」だ。

### AIエージェントがNHIの数を爆発させる

従来のNHIはCI/CDパイプラインやマイクロサービスの数に比例して増えた。しかしAIエージェントの時代には、**一人の開発者が一日に何十ものセッションを生成する。** Claude Codeを使う開発者が10人いれば、毎日数百のセッション——それぞれが独立したNHIとして扱われるべきID——が生まれる。

しかしこれらは現状、IDとして管理されていない。

---

## Claude CodeセッションはNHIである

ここが記事の核心だ。Claude Codeのセッションが「ただのツール」ではなく、独立したNHIとして管理されるべき理由を整理する。

### NHIとしての性質

Claude Codeセッションは以下の性質を持つ。

- **独自のAPIコンテキスト**を保持する（anthropic_session_idに相当するコンテキスト識別子）
- **ファイルシステム・ターミナル・ネットワーク**へのアクセス権を持つ
- 人間の監視なしに**長時間の自律実行**が可能（Computer Useやサブエージェント委譲）
- **複数の外部APIキー**にアクセスできる（.envファイル、シークレットマネージャー経由）

これは典型的なサービスアカウントの性質と同一だ。むしろ、従来のサービスアカウントよりも**動的で広範なアクセスパターン**を持つ。

### しかし現状は管理されていない

| 管理項目 | 人間の開発者 | Claude Codeセッション（現状） |
|---------|------------|---------------------------|
| アイデンティティ | 個人アカウント（SSO/IdP管理） | APIキーを共用（セッション単位の識別なし） |
| 権限のスコープ | ロール・グループで明示定義 | 実質的に無制限（ツール側が許可する限り） |
| 監査ログ | ログイン履歴・操作ログが組織に残る | ローカルの~/.claude/logsのみ |
| 権限のライフサイクル | 入退社・異動で見直し | セッション終了後も認証情報は有効のまま |
| インシデント対応 | アカウントロック・強制ログアウト | セッションを止める仕組みがない |

この表を見れば、Claude Codeセッションが**人間ユーザーよりはるかに低いガバナンス水準**で動いていることがわかる。

## NHIの4原則をClaude Codeに適用する

OktaとCyberArkがNHI管理フレームワークとして提示している原則は4つある。それぞれをClaude Codeに具体的に適用してみよう。

### 原則1：最小権限（Least Privilege）

NHIには「そのタスクに必要な最小限の権限のみ」を与える。Claude Codeであれば、プロジェクトルートの`CLAUDE.md`またはポリシーファイルで明示的に定義する。

```yaml
# claude_code_policy.yaml
# Claude Codeセッションに付与する権限の明示的な定義

agent_permissions:
  file_access:
    - path: "./src/**"
      mode: read_write
    - path: "./tests/**"
      mode: read_write
    - path: "./docs/**"
      mode: read_only
    - path: ".env*"
      mode: deny        # 環境変数ファイルへのアクセス禁止
    - path: "*.pem"
      mode: deny        # 証明書ファイルへのアクセス禁止

  shell_commands:
    allowed:
      - "npm test"
      - "npm run build"
      - "git status"
      - "git diff"
      - "git log"
    denied:
      - "rm -rf *"      # 破壊的削除の禁止
      - "curl *"        # 任意の外部通信の禁止
      - "wget *"
      - "ssh *"

  network_access:
    allowed_hosts:
      - "api.anthropic.com"
      - "registry.npmjs.org"
    denied_hosts:
      - "*"             # 上記以外の全ホストへの通信を拒否（デフォルト拒否）
```

`CLAUDE.md`で同等のルールを自然言語で定義することもできるが、YAML形式のポリシーファイルは**機械的に検証可能**という利点がある。

### 原則2：一時的アクセス（Just-in-Time）

NHIの認証情報は「必要なときに発行し、不要になったら即失効」が原則だ。人間の開発者のAPIキーとClaude Code専用のAPIキーを分けるだけでなく、**タスク単位の一時トークン**の発行を目指す。

現実的な第一歩は以下だ：

- `ANTHROPIC_API_KEY`をClaude Code専用のキーに分離し、他のサービスと共用しない
- Claude Codeセッション開始時にスクリプトで一時的な読み取り専用コンテキストを設定し、終了時に解除する
- AWSやGCPを使っている場合、STSの一時クレデンシャルやWorkload Identity Federationを活用してClaude Codeセッションにスコープされた権限を発行する

### 原則3：監査可能性（Auditability）

「いつ・どのセッションが・どのリソースに・何をしたか」を後から追跡できる状態にする。Claude Codeのデフォルトログ（`~/.claude/logs/`）は開発者のローカルにしか残らない。組織としての監査には不十分だ。

実践的な対応：

- Claude Codeのフック機能（`PreToolUse`/`PostToolUse`）を使い、全ツール呼び出しを組織の中央ログ基盤（CloudWatch、Datadog等）に送信する
- 少なくとも「ファイル書き込み」「シェル実行」「外部ネットワーク接続」の3カテゴリは記録する
- セッションIDと開発者の識別子を紐付けて記録することで、インシデント時の追跡を可能にする

### 原則4：アイデンティティの分離（Identity Isolation）

開発者AのClaude Codeと開発者BのClaude Codeは、同じAPIキーで動いていても**別のNHI**として扱うべきだ。現状は「チーム全員が同じ`ANTHROPIC_API_KEY`を使っている」ケースが多いが、これは一人のサービスアカウントをチーム全員で共用しているのと同義だ。

実現方法：

- 開発者ごとに個別のAPIキーを発行し、使用量と操作履歴を個人に紐付ける
- エンタープライズ環境ではAnthropicのWorkspaces機能を活用してチーム単位のAPIキー管理を行う
- SSO連携でClaude Codeへのアクセスを組織のIdP（Okta等）経由に統一する

## NHI管理フレームワーク：ライフサイクル全体の比較

NHI管理は「一度設定して終わり」ではない。人間のIAMと同様に、ライフサイクル全体を通じた管理が必要だ。

| ライフサイクル | 人間ユーザー（従来のIAM） | AIエージェント（NHI管理） |
|--------------|------------------------|------------------------|
| プロビジョニング | HR入社手続きでアカウント作成、初期ロール付与 | タスク開始時にスコープされたセッション認証情報を発行 |
| 権限管理 | ロール・グループで制御、定期アクセスレビュー | ポリシーファイルでツール・パス・コマンドを制御 |
| 監査 | ログイン履歴・VPN接続ログ・操作ログ | ツール呼び出しログ・コマンド実行ログ・ファイルアクセスログ |
| クレデンシャルローテーション | 90日パスワード変更ポリシー | セッション終了ごとに一時認証情報を失効、定期的なAPIキーローテーション |
| デプロビジョニング | 退職時にアカウントを無効化、グループから削除 | セッション終了時に一時認証情報を失効、プロジェクト終了時にAPIキーを削除 |
| インシデント対応 | アカウントロック、強制ログアウト、パスワードリセット | セッション強制終了、APIキー即時失効、アクセス経路の遮断 |
| 棚卸し・ガバナンス | 四半期ごとのアクセスレビュー | 稼働中エージェントのインベントリ作成と定期レビュー |

この表で重要なのは、**AIエージェントのライフサイクルは人間より短く・高速**だという点だ。人間の「四半期レビュー」が、エージェントでは「セッション単位の自動失効」に相当する。管理の粒度が異なる。

## 企業が今すぐできること

理想的なNHI管理基盤の構築には時間がかかる。だが以下の4ステップは今日から着手できる。

### Step 1：インベントリ作成

社内で動いているClaude Codeセッションの数と、それぞれが使っているAPIキーを把握する。まず「何が動いているかわからない」状態から脱することが最優先だ。開発者にアンケートを取るだけでもよい。

### Step 2：APIキーのスコープ分離

Claude Code専用のAPIキーを発行し、他サービスのキーと完全に分離する。同じキーがClaude Code・CI/CD・バックエンドAPIで使い回されている場合、Claude Codeのセッションが侵害されれば全サービスへの影響が波及する。

### Step 3：CLAUDE.mdでのルール明示（ポリシー・アズ・コード）

各プロジェクトの`CLAUDE.md`に、Claude Codeセッションが行ってよいこと・してはいけないことを明文化する。これは人間向けのメモではなく、**エージェントのガバナンスポリシーをコードとして管理する**という位置づけで書く。

```markdown
# CLAUDE.md（ガバナンスポリシーセクションの例）

## セキュリティポリシー

- .env ファイル・*.pem・シークレット関連ファイルは読み取り・変更・削除禁止
- `curl` `wget` `ssh` コマンドの実行前に必ずユーザーに確認すること
- `rm -rf` の実行は禁止
- 外部APIへのリクエストは事前に目的を説明し、承認を得ること
```

### Step 4：ログの集中管理

各開発者のローカルに散らばっているClaude Codeのログを、組織の中央ログ基盤に集約する仕組みを作る。Claude Codeのフック機能を使えば、`PostToolUse`フックでツール実行結果を外部エンドポイントに送信できる。

## まとめ

NHIという概念は難しくない。「人間以外のものがシステムにアクセスするとき、そのアクセスをどう管理するか」という問いだ。

サービスアカウントやAPIキーの管理に取り組んできた組織なら、その延長線上にClaude Codeセッションを置くだけでよい。違いは**速度と数**だ。従来のNHIは数週間単位で増減したが、AIエージェントは一日に何百ものセッションを生成する。

Verizon DBIRが「侵害の68%がNHIの不正利用から」と報告している現状で、最も管理が遅れているカテゴリ——AIエージェントセッション——を放置することのリスクは明確だ。

「AIエージェントのIDを管理しているか？」という問いへの答えが「NO」なら、今日からインベントリの作成を始めるのが最初の一歩だ。
