## Option A — Text post, self-audit framing (recommended)

**Title:** `Tell HN: I audited my own AI security OSS, published the findings in the release notes`

**Body:**

```
Context: I maintain Aigis, an OSS firewall for AI agents. Just shipped v0.0.4.

This release is a deliberate experiment in what a security-product release note should look like. The product itself is 940+60 tests passing, pure-Python, zero core deps. But it's a security product, so "my product is secure" is a claim that needs to be verifiable, not just stated.

For v0.0.4 I did a structured self-audit of the backend + scanner core and documented every finding directly in the release notes:

- 4 Critical: unsalted SHA-256 + `==` API key verify, `*`+credentials CORS, Stripe webhook without idempotency, unauth'd tenant-creation endpoint
- 5 High: Slack webhook SSRF, JWT algo/claim pinning gap, ReDoS in custom_rules with silent error swallowing, confusables coverage gaps, docs exposure layering
- 4 Medium: SECRET_KEY/DB placeholder validators, bcrypt rounds pinning, misc
- 7 CodeQL alerts (6× `py/overly-permissive-regex-range`, 1× clear-text sensitive logging)

All fixed, all in the CHANGELOG with the mechanism, the exploitable path, and the concrete fix: https://github.com/killertcell428/aigis/releases/tag/v0.0.4

The reason I'm posting this rather than quietly patching: a security OSS that hides its own audit findings is worse than one that publishes them. Disclosure is part of the product.

Separate track in the same release: seven 2025-2026 LLM-security papers implemented as one module each, zero extra deps. Every module was mapped onto the existing 4-wall + L4-L6 defence layout rather than bolted on as a side section. That's where the design work was.

If anyone here has a self-audit disclosure template they've used successfully for defensive-security OSS, I'd like to compare notes. Happy to share my current format.
```
