## 2. ターゲット戦略

### ファネル構造

```
認知（Qiita/Zenn/Twitter → Gandalf Challenge）
  ↓ ペルソナ1（エンジニア）が触る
試用（pip install → aig init → 動作確認）
  ↓ 社内提案テンプレートで情シスに持っていく
承認（ペルソナ2がaig status / aig reportを見てOK出す）
  ↓
導入（チーム全体にaig init）
  ↓
拡大（ペルソナ3が経営会議でAIガバナンス報告 → 全社展開）
```

### キーインサイト
- **売る相手はエンジニア（田中）、買う人は情シス（鈴木）、決める人は経営（佐藤）**
- エンジニア向けにはQiita/Zennの技術記事
- 情シス向けには「社内提案テンプレート」と「Excel出力」
- 経営向けには「コンプライアンス100%」の数字

## 3. ネーミング検討（商品ネーミングフレームワーク準拠）

### 現状：「Aigis」

**評価:**
| 原則 | 評価 | 備考 |
|------|------|------|
| 読みやすさ | ○ | 「エーアイ・ガーディアン」明快 |
| 独自性 | △ | 「Aigis」はかなり一般的。同名ツール・サービスが複数存在 |
| 語感と意味の一致 | ○ | 「守る」イメージは適切 |
| 拡張性 | △ | 「Guardian」は防御のイメージ。攻めのガバナンスプラットフォームには狭い |

### 問題点
- 「Aigis」でググると他社製品がヒットする可能性
- 「ガーディアン=守り」のイメージが強く、「可視化・制御・証明」のプラットフォーム感が薄い
- v2の「エージェントガバナンスハブ」という新ポジションには合わなくなりつつある

### 代替案（ネーミング3類型 × 表現方法3タイプ）

#### 連結型 × 機能重視
| 候補 | 構成 | イメージ |
|------|------|---------|
| **AigisOS** | Aigis(盾の女神) + OS | エージェントの共通OS。神話由来で独自性高い |
| **AgentShield** | Agent + Shield | 直球。わかりやすいが独自性は低い |
| **GuardStack** | Guard + Stack | 「スタック」で多層防御を表現 |

#### 連結型 × イメージ重視
| 候補 | 構成 | イメージ |
|------|------|---------|
| **Aegis** | ギリシャ神話の盾 | 格好いい。だが既に使われすぎ |
| **Bastion** | 要塞 | 堅牢さ。だがやや重い |
| **Sentris** | Sentinel + Iris | 見張り+虹彩（監視の目）。造語で独自性◎ |

#### 混合型 × 機能重視
| 候補 | 構成 | イメージ |
|------|------|---------|
| **AIG** | AI + Governance の頭文字 | シンプル。CLIの`aig`コマンドとも一致。ただしAIG(保険)と被る |
| **Govana** | Governance + Ana(分析) | ガバナンス分析プラットフォーム。柔らかい響き |
| **Patronus** | ラテン語「守護者」 | ハリポタ連想で親しみやすい |

### 推奨：現時点では「Aigis」を維持

**理由:**
1. CLIコマンド `aig` が既に定着（Aigisの頭文字）
2. PyPI `aigis` で公開済み
3. Vercelサイト稼働中
4. 変更コストが高い（GitHub, PyPI, サイト, 記事ドラフト全て）
5. 「Aigis」は十分わかりやすく、SEOでも「AI agent guardian governance」で差別化可能

**将来的なリネーミングのタイミング:**
- PMFが確認できて本格展開する時
- シリーズAなどの資金調達時
- 競合との名前被りが問題になった時
